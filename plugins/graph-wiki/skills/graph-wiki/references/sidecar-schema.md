# `<vault>/work-index.json` — schema reference

Generated by `/graph-wiki:regen-index`. Read by
`/graph-wiki:status` and any other
consumer that needs structured access to the work queue.

**Never edit by hand** — it's regenerated from `<vault>/work/*.md`.

> **Note:** The work-layer subsystem (regen-index, status) is not ported in graph-wiki v1.2. This reference doc is retained for parity with upstream; the schema applies when/if work-layer support is added in a future version.

## Top-level shape

```json
{
  "schema_version": 1,
  "generated_at": "ISO-8601 UTC, Z suffix",
  "vault_commit": "git HEAD of the vault repo, or null",
  "counts": { ... },
  "items": [ ... ]
}
```

## `counts` — denormalized rollup

Always present, all keys always present, zeroed when no items match.

```json
{
  "by_status":       {"open": N, "accepted": N, "in-progress": N, "mitigated": N, "resolved": N, "wontfix": N, "superseded": N},
  "by_kind":         {"bug": N, "tech-debt": N, "test-gap": N, "security": N, "perf": N, "feature": N, "initiative": N, "spike": N},
  "by_severity":     {"low": N, "medium": N, "high": N, "critical": N},
  "by_blast_radius": {"file": N, "package": N, "domain": N, "system": N}
}
```

`by_severity` only counts items where `severity:` is set — `feature`,
`initiative`, and `spike` typically don't contribute. This is by design.

## `items` — per-item digest

Ordered by `opened` descending, then `slug` ascending. Stable: the same
input produces the same JSON.

| Field | Type | When null |
|---|---|---|
| `slug` | string | never |
| `title` | string | never (empty string if absent in frontmatter) |
| `kind` | string | never (one of the 8 enum values) |
| `status` | string | never (one of the 7 enum values) |
| `severity` | string \| null | when frontmatter `severity:` empty |
| `effort` | string \| null | when frontmatter `effort:` empty |
| `blast_radius` | string \| null | when frontmatter `blast_radius:` empty |
| `affects` | array of strings | always array, possibly empty |
| `target` | string \| null | when frontmatter `target:` empty |
| `owner` | string \| null | when frontmatter `owner:` empty |
| `plan_steps` | integer | never (0 when `## Plan` absent or unparseable) |
| `opened` | string (ISO date) \| null | when frontmatter `opened:` empty |
| `updated` | string (ISO date) \| null | when frontmatter `updated:` empty |
| `resolved_in` | string \| null | when status is not 'resolved' |
| `superseded_by` | string \| null | when status is not 'superseded' |
| `mitigation` | string \| null | when status is not 'mitigated' |
| `rationale` | string \| null | when status is not 'wontfix' |
| `pr` | string \| null | when frontmatter `pr:` empty |
| `branch` | string \| null | when frontmatter `branch:` empty |
| `tags` | array of strings | always array, possibly empty |

## Freshness contract

Any `items[*].updated > generated_at` => sidecar is stale => regenerate.

Consumers MUST check before relying on the data. The reference Python
helper (in `lib/sidecar.py`) is three lines — vendor or re-implement:

```python
def is_stale(sidecar: dict) -> bool:
    generated_date = sidecar.get("generated_at", "")[:10]
    return any((i.get("updated") or "") > generated_date for i in sidecar.get("items", []))
```

## What's NOT in the v1 sidecar

- Cross-reference rollups ("which ADRs reference this item") — open
  decision 2.4-E; deferred until consumer query patterns are observable.
- Graph-aware `affects:` resolution — deferred to v1.1; per ADR-W3 of the
  v1 design, filesystem resolution is the v1 contract.

If the sidecar gains a `schema_version: 2`, the work-tracker will ship a
migrator at `scripts/migrate_v1_to_v2.py`.
