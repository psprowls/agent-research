# Quick Task 260521-lj3 Plan 01: workspace-io tolerate missing plugins key — Summary

One-liner: `init()` now heals sparse `.graph-wiki.yaml` manifests in place via `data.setdefault("plugins", [])`, unblocking `/graph-wiki:bootstrap` after the documented provisional-manifest workaround.

## What Was Built

A one-line defensive fix plus a regression test:

- `packages/workspace-io/src/workspace_io/init.py` — inserted `data.setdefault("plugins", [])` between the load block (`if mpath.exists(): data = manifest.read(mpath) else: ...`) and the `entry = next(...)` lookup. The line heals manifests that contain `version: 2` (plus arbitrary other keys like `repo:`) but lack the top-level `plugins` list. `manifest.write()` already uses `data.get("plugins", [])` so the round-trip was already safe; only the lookup needed defending.
- `packages/workspace-io/tests/test_init.py` — added `test_init_tolerates_existing_manifest_without_plugins_key`, which pre-writes a literal `version: 2\nrepo: /some/path\n` manifest, calls `init(...)`, then asserts the resulting `data["plugins"]` is a 1-entry list with the correct `name` / `installed_version` / `applied_version`.

## Why

The `/graph-wiki:bootstrap` flow can seed a provisional `.graph-wiki.yaml` to satisfy the `resolve_wiki_and_repo()` chicken-and-egg in `detect_containers.py` — that provisional manifest legitimately lacks the `plugins` key. The subsequent `init()` call then exploded on `data["plugins"]` with `KeyError: 'plugins'`. The plan deliberately scoped the fix to the write path (`init.py`) so the canonical v2 schema (`plugins` required) is unchanged — this is purely defensive against a known transient on-disk state.
