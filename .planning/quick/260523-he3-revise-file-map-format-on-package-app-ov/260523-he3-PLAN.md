<objective>
Revise the package/app overview "File map" format from H2 + paragraph + bullets + H3-H6 nested sub-sections to H2 + per-major-folder markdown tables. Lockstep change across emitter, parser, templates, reference docs, agent prompt, and tests. The parser will be replaced (not dual-mode) with a graceful fallback (returns empty list) when handed old-format content so existing wiki pages on disk don't trigger false-positive drift before they're re-emitted on the next scan.

Purpose: Tables are easier to scan visually, easier to fill in (one row at a time vs. nesting state), and remove the H4/H5/H6 ladder that was awkward in Obsidian rendering. Aligns with the existing convention for the `## Routes / screens` and `## Versions in use` tables already used elsewhere in the page-formats spec.

Output:
- Rewritten `build_file_map()` and `parse_section_entries()`
- Updated templates (`package/overview.md`, `app.md`)
- Updated reference docs (`page-formats.md`, `scan-workflow.md`, `lint-workflow.md`)
- Updated `scanner.md` agent prompt
- Updated tests (snapshot-style + drift-detection cases for new format; one fixture for old-format graceful fallback)
- Green test suite under `packages/wiki-io/`
</objective>

<new_format_spec>
Below is the canonical shape every other artifact in this plan must conform to. Both `build_file_map()` output and the templates must produce exactly this structure.

```markdown
## File map - <pkg-name>
One-sentence overview of what this package's tree contains.

### <pkg-name>/
Files at the package root.

| Path | Kind | Description |
|---|---|---|
| `package.json` | file | — TODO |
| `README.md` | file | — TODO |

### <pkg-name>/src/
TypeScript source.

| Path | Kind | Description |
|---|---|---|
| `index.ts` | file | — TODO |
| `handlers/baseApiHandler.ts` | file | — TODO |
| `handlers/withGlobalContext.ts` | file | — TODO |
| `middleware/authProvider.ts` | file | — TODO |
| `clients/` | dir | — TODO |

### <pkg-name>/tests/
Tests.

| Path | Kind | Description |
|---|---|---|
| `handlers.test.ts` | file | — TODO |
```

Rules:
- One H3 section per depth-1 entry plus one synthetic `### <pkg-name>/` for root-level files. Sort H3 sections alphabetically; root section first.
- Each H3 section: H3 heading, one-line paragraph (placeholder `TODO — describe what this directory contains.` initially), markdown table.
- Table columns are exactly `Path | Kind | Description` with separator `|---|---|---|`.
- `Path` is relative to the H3's root. For `### <pkg>/src/`, paths are `index.ts`, `handlers/baseApiHandler.ts` (no `src/` prefix, no leading slash). For `### <pkg>/` (root), paths are bare filenames or first-level dir names.
- `Kind` is exactly `file` or `dir`.
- `Description` starts as `— TODO`.
- Files deeper than depth 1 flatten into rows inside their depth-1 parent's table (e.g. `middleware/authProvider.ts` lives in the `### <pkg>/src/` table). Directories deeper than the `max_depth` cutoff (default 4) are listed as `dir` rows in their parent's depth-1 table instead of getting their own section.
- Path tokens are wrapped in backticks. Directory paths end with `/`. File paths do not.
- The truncation marker (`> Truncated at N files.`) remains a blockquote at the bottom of the whole block, after the last H3 section.
</new_format_spec>

```
## File map convention (apps and packages)

App and package pages have a `## File map - <name>` section composed of one H3 subsection per major folder, each containing a markdown table. Rules:

- The H2 heading carries the package or app name: `## File map - <name>`, followed by a one-line overview paragraph.
- Files at the workspace root live in a synthetic `### <name>/` H3 section directly under the H2 — uniform shape, no special-cased root.
- Each depth-1 subdirectory gets its own H3 section whose heading is the full path from the workspace root with a trailing slash (e.g. `### <name>/<sub>/`).
- Under each H3: a one-sentence paragraph describing the directory, then a markdown table with columns `Path | Kind | Description`. `Path` is relative to that section's root (e.g. `middleware/auth.ts` inside `### <name>/src/`). `Kind` is `file` or `dir`. `Description` starts as `— TODO` and is filled in by the agent later.
- Nested files (depth ≥ 2) flatten into rows inside their depth-1 parent's table. Directories deeper than the cutoff (default `max_depth=4`) are listed as `dir` rows in their depth-1 parent's table instead of getting their own section.
- The scanner pre-populates the tables via `git ls-files` (so `.gitignore` is respected) with `— TODO` Description placeholders. Per-row descriptions are filled in by the agent on a later pass.
- `/graph-wiki:lint`'s file-map drift check flags rows whose Path is no longer on disk; new files showing up on disk do not fail lint, since `dir`-row summarization is allowed.
- **Legacy heading+bullet pages on disk** (pre-2026-05) are parsed gracefully: directory entries from H3 headers are still extracted, but file-row entries are dropped. The next scan re-emits the block in the new table format when the page still shows the unfilled-template signature.
```
