<split_rule_spec>
Single source of truth for the prod vs testing classification used by the emitter, the templates, and the docs. Every artifact in this plan must agree with this spec.

A file path (relative to the package root) is classified as **test** iff at least one is true:

1. **Directory match**: any path component (case-sensitive) is exactly one of:
   - `tests`
   - `__tests__`
   - `test`
   - `spec`
   So `tests/handlers.test.ts`, `src/__tests__/foo.spec.ts`, `packages/x/test/a.py`, and `cypress/spec/login.cy.ts` are all test files. A single component named `tested.ts` is NOT a match (exact directory-name match only).

2. **Test-config basename match**: the path is exactly one component long (root-level file) AND the basename matches one of:
   - `conftest.py`
   - `pytest.ini`
   - `tox.ini`
   - `pyproject-tests.toml` (rare; include for symmetry)
   - `jest.config.js`, `jest.config.ts`, `jest.config.mjs`, `jest.config.cjs`, `jest.config.json`
   - `vitest.config.js`, `vitest.config.ts`, `vitest.config.mjs`
   - `playwright.config.js`, `playwright.config.ts`
   - `cypress.config.js`, `cypress.config.ts`
   - `mocha.config.js`, `.mocharc.js`, `.mocharc.json`, `.mocharc.yaml`, `.mocharc.yml`
   - `karma.conf.js`, `karma.conf.ts`
   - `ava.config.js`, `ava.config.cjs`, `ava.config.mjs`
   Implemented as a fixed set + a compiled regex for the `*.config.<ext>` variants — see the emitter's `_TEST_CONFIG_NAMES` set and `_TEST_CONFIG_RE` pattern below.

All other paths are **prod**. Files that are deeper than max_depth keep their existing summarization (collapse to a `dir` row in their depth-1 parent's table), but the classification of the parent determines which file_map block the row lands in: a `tests/<deep>/<file>` row goes into the testing block under `### <pkg>/tests/`; a `src/<deep>/<file>` row goes into the overview block under `### <pkg>/src/`.

A pkg with zero test files emits a placeholder testing block (synthetic `### <pkg>/` H3 with the no-test-files marker — see emitter spec, Task 2). The agent still creates `testing.md` from the template; the file_map block on it just declares "no tests yet". This keeps the wiki layout uniform across packages regardless of test coverage.

A pkg with zero prod files (extremely rare — a tests-only meta-package) still produces a normal overview block via the existing empty-package short circuit (`- (no tracked files)` line). No special case needed.
</split_rule_spec>

<emitter_contract>
Below is the contract `build_file_map()` must satisfy after Task 2. The current single-return signature stays for backward compatibility; a new paired API is added.

Current (unchanged, kept for back-compat):
```python
def build_file_map(pkg_path: Path, max_depth: int = 4, max_entries: int = 80) -> str | None:
    """Returns one combined block (prod + tests interleaved) — preserved as the
    one-shot API for callers that don't want the split. Internally, this calls
    build_file_maps() and joins the two blocks."""
```

New (paired API, what Task 2 adds):
```python
def build_file_maps(
    pkg_path: Path,
    max_depth: int = 4,
    max_entries: int = 80,
) -> tuple[str, str] | None:
    """Return ``(prod_block, test_block)`` where each is a full markdown
    ``## File map - <name>`` block in the table format established by
    quick-260523-he3. The split rule lives in ``_is_test_path()`` and is the
    single source of truth.

    - prod_block: H2 + per-major-folder H3 sections covering ONLY prod files +
      prod root-level config. Test directories (tests/, __tests__/, test/,
      spec/) and test-config files are excluded.
    - test_block: H2 + per-major-folder H3 sections covering ONLY test files,
      test config (at any path), and fixtures (rows under tests/fixtures/ are
      naturally included since their parent component matches the dir rule).

    Returns None when ``_git_ls_files(pkg_path)`` returns None, matching the
    existing build_file_map() contract.

    When there are no test files in the package, ``test_block`` is a minimal
    placeholder: H2 + a single ``### <pkg>/`` synthetic root H3 + a paragraph
    ``TODO — no test files detected. Document test strategy here when tests
    land.`` and no table.

    When there are no prod files (extremely rare), ``prod_block`` uses the
    existing empty-package short circuit (``- (no tracked files)``).
    """
```

The existing one-shot `build_file_map()` is rewired to call `build_file_maps()` and return the prod_block alone (NOT a concatenation — concatenation would re-mix prod and test rows). Callers that want both must use the new API. This keeps backward compatibility for any external consumer while making the split the canonical shape going forward.

The CLI integration in `scan_monorepo.py main()` (around line 1221-1226) is updated to call `build_file_maps()` and put both blocks into the workspace dict:
```python
fms = build_file_maps(pkg_dir, max_depth=args.max_depth)
if fms is not None:
    prod_block, test_block = fms
    w["file_map"] = prod_block
    w["file_map_testing"] = test_block
```

**Q2 — Plugins have no existing template (RESOLVED: clone from package, then specialize lightly.)**
Today `_collect_claude_plugin()` returns a workspace dict; the scanner agent uses `package/` templates for plugins by routing through `ensure_subpage(pkg_dir, ..., "package", ...)` (or in practice, by treating plugins as packages). Create `assets/page-templates/plugin/overview.md` as a near-copy of `package/overview.md` with `category: plugin` and a plugin-specific Purpose blurb. Adding the templates only changes what `ensure_subpage` can find when the agent eventually asks for `plugin/testing.md`.

**Q3 — Where do fixtures live? (RESOLVED: under tests/, classify as test path.)**
The split rule's directory match handles all the listed test directory names. Fixtures under `tests/fixtures/` are naturally captured by the `tests` component. A workspace that puts fixtures at a non-test path (e.g. `fixtures/` at the root with no `tests/` prefix) will have them classified as prod — that's a deliberate choice: such fixtures double as runtime data in many projects (golden files, snapshots, demo data). If a project standardizes on root-level `fixtures/`, the maintainer can document them in `testing.md`'s ## Fixtures section by hand; the file_map doesn't auto-classify them. Documented in page-formats.md.

**Q4 — Should `## File map - <pkg>` H2 heading text differ between overview and testing? (RESOLVED: same heading, different content.)**
Both pages use `## File map - <pkg-name>` (not `## File map - <pkg-name>-tests`). Rationale: SCN-003 in eval-harness asserts the scanner's stub output does not contain the substring `## File map`. Keeping the heading identical (the substring `## File map` is invariant) means SCN-003 keeps its current behavior — no eval-harness change required, just like in 260523-he3. The page identity is established by the file path (`<pkg>/overview.md` vs `<pkg>/testing.md`), not the heading text.

**Q5 — Should the legacy `parse_section_entries()` parser handle two file_maps per workspace? (RESOLVED: no, parser is per-page.)**
`parse_section_entries()` already operates on the body of a single `## File map - <pkg>` section. It doesn't care which page it's on. After this change, the same parser runs against both overview.md and testing.md and produces drift findings against the union of rows. No parser change is required.

**Q6 — Sub-pages list on overview.md (RESOLVED: append `testing`.)**
The `## Sub-pages` block in `package/overview.md` currently lists `api`, `patterns`, `work`, `context`. Add a fifth row for `testing` with the same `[[wiki/{{CONTAINER_DIR}}/{{PACKAGE_SLUG}}/testing|testing]]` wikilink form. Order: `testing` goes after `work` and before `context` so the cognitive grouping is (api → patterns → work → testing → context) — implementation surface first, then process surface. `app/overview.md` and `plugin/overview.md` (created in Task 1) start with the same five-item list.

**Q7 — Drop empty placeholder testing.md when a package has no tests? (RESOLVED: no, always create.)**
Every package gets a testing.md from the template, even if `build_file_maps()` returns a no-tests-detected placeholder block. The page acts as a forcing function: when a maintainer adds tests, the placeholder is replaced by the next scan. This matches the existing pattern of always-scaffolded sub-pages (api.md, patterns.md, etc., are written regardless of whether the package has a public API to document).

_TEST_CONFIG_RE = re.compile(
    r"^("
    r"jest\.config\.(?:js|ts|mjs|cjs|json)"
    r"|vitest\.config\.(?:js|ts|mjs)"
    r"|playwright\.config\.(?:js|ts)"

def _is_test_path(rel: str) -> bool:
    """Classify a package-relative path as test (True) or prod (False).

    Rule:
      1. Any path component (split on '/') matching _TEST_DIR_NAMES → True.
      2. Basename in _TEST_CONFIG_NAMES OR matching _TEST_CONFIG_RE → True
         (applies at any depth; conftest.py is a common pytest pattern at
         non-root paths too).
      3. Otherwise → False.

    Precondition: ``rel`` is a forward-slash-separated path relative to the
    package root (the same shape build_file_map already receives from
    _git_ls_files).

    if any(p in _TEST_DIR_NAMES for p in parts):
        return True
    basename = parts[-1]
    if basename in _TEST_CONFIG_NAMES or _TEST_CONFIG_RE.match(basename):
        return True
    return False
