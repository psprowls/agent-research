<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP host (subprocess parent) → MCP server (subprocess child) | Untrusted JSON-RPC messages cross stdin into the server; FastMCP validates against the registered Pydantic schema |
| process → stdout | Stdout is the JSON-RPC channel; any non-frame byte corrupts the protocol |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-02 | Tampering | MCP stdout / JSON-RPC framing | mitigate | `_StdoutGuard` rebinds `sys.stdout` at module-init time so any non-FastMCP write raises immediately. `logging.basicConfig(stream=sys.stderr)` routes all logging output away from stdout. The subprocess test in Task 2 is the end-to-end proof: every stdout line is `json.loads`-able with a `jsonrpc` or `id` key. This is the PRIMARY mitigation for MCP-05. |
| T-1-08 | Input Validation | wiki_ping input | mitigate | `PingInput(BaseModel)` validates that `message` is a string. FastMCP rejects malformed inputs with a structured error response (no crash, no stdio session kill — matches MCP-04 spirit even though MCP-04 is Phase 3). |
| T-1-09 | Information Disclosure | Server error messages | accept | Phase 1 has no sensitive data; wiki_ping returns the user's own input. Future tools that touch real vault data will revisit. |
</threat_model>
