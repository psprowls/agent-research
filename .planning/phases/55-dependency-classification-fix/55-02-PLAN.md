<objective>
Make `cg describe-package <name>` surface both directions of the `depends_on_package` edge
introduced in Plan 01: internal DEPENDENTS (incoming — workspace packages that depend on this
one, mandatory per SC#3) and internal DEPENDENCIES (outgoing — workspace packages this one
depends on, the natural complement per D-08). This is the query/CLI half of CLASS-02 — Plan 01
emits the edge, this plan reads it back.

Purpose: SC#3 requires the internal package→package relationship to "surface correctly in graph
queries (e.g. `cg describe-package <name>` shows internal dependents)". Today `describe_package()`
has no awareness of `depends_on_package` edges, so the relationship Plan 01 emits would be
invisible at the query surface.

Output: `describe_package()` returns two new collections, exposed in both JSON
(`dataclasses.asdict`) and the CLI text output; query + CLI tests assert both directions.
</objective>

## Trust Boundaries

This plan touches only `graph-io`'s read-only query layer (`describe_package`) and its CLI
rendering. Queries run through `store.read_only_connect()` against the local sqlite graph; the
CLI prints to stdout. No writes, no network, no untrusted input, no new execution surface.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-02-T1 | Information Disclosure | A wrong JOIN direction swaps dependents/dependencies, misreporting the graph relationship | mitigate | Task 3 asserts BOTH directions explicitly (incoming `internal_dependents` = the consumer; outgoing `internal_dependencies` = the target) against a single known edge, so a swapped JOIN fails the test. |
| T-55-02-T2 | Tampering | SQL built by string-interpolating the package name (injection / wrong match) | mitigate | The new queries use `?` placeholders only (Task 1 action mandates it), mirroring the existing `describe_package` parameterized queries. |
