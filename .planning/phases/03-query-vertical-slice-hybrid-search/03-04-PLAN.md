Pydantic schemas to add to server.py:
  class WikiQueryInput(BaseModel):
      query: str
      vault_path: str = ""    # empty => resolve from CODE_WIKI_REAL_VAULT_PATH env var
      top_k: int = 5          # 3-10 range (validated by run_query)

  class WikiQueryOutput(BaseModel):
      answer: str
      citations: list[str]
      pages_drilled: int
      search_scores: dict     # {page_path: {"bm25": float, "embed": float, "rrf": float}}

Tool registration:
  @mcp.tool(
      name="wiki_query",
      description=(
          "Query the code wiki using hybrid BM25+embedding search with parallel librarian "
          "analysis. Returns an answer with [[wikilink]] citations. "
          "vault_path defaults to CODE_WIKI_REAL_VAULT_PATH env var."
      ),
  )
  async def wiki_query(input: WikiQueryInput, ctx: Context) -> WikiQueryOutput:
      vault = Path(input.vault_path) if input.vault_path else None
      await ctx.report_progress(progress=0, total=input.top_k, message="Starting hybrid search")
      result: QueryResult = await run_query(
          query=input.query, vault_path=vault, top_k=input.top_k,
      )
      await ctx.report_progress(
          progress=result.pages_drilled, total=input.top_k,
          message=f"Synthesized from {result.pages_drilled} pages",
      )
      return WikiQueryOutput(
          answer=result.answer,
          citations=result.citations,
          pages_drilled=result.pages_drilled,
          search_scores=result.search_scores,
      )

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP wire | JSON-RPC over stdio; FastMCP validates `WikiQueryInput` Pydantic schema at boundary |
| `vault_path` argument from MCP client | Untrusted path input — must be resolved via `resolve_wiki_and_repo` which calls `Path.resolve()` |
| Subprocess test environment | `uv run code-wiki-mcp` launches a child process; stdout must remain pure JSON-RPC |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Tampering | Path traversal via `WikiQueryInput.vault_path` (HIGH per planning context) | mitigate | `wiki_query` passes the path to `run_query` → `resolve_wiki_and_repo` which calls `Path.resolve()`; resolved path is then used for all reads via `_discover_pages` (rooted at vault). Pydantic field `vault_path: str = ""` is bound — no shell expansion. Plan 03 Task 1 verifies via test_apply_guardrails the read path stays within vault. |
| T-03-14 | Spoofing | Malformed JSON-RPC input crashing the MCP session (MCP-04) | mitigate | Pydantic `WikiQueryInput` validates at boundary; `Field(ge=3, le=10)` enforces top_k range; FastMCP wraps Pydantic ValidationError into a structured MCP error response — does not crash the stdio session. Verified by `test_wiki_query_input_rejects_*` unit tests and by the existing `_StdoutGuard` ensuring nothing leaks to stdout outside JSON-RPC framing. |
| T-03-15 | Information Disclosure | Server stdout leaking non-JSON content during error paths | mitigate | `_StdoutGuard` (Phase 1 deliverable) raises on any non-empty stdout write; integration test `test_wiki_query_in_tools_list` verifies tools/list returns valid JSON, confirming the guard remains active with the new tool added. |
| T-03-16 | Denial of Service | Oversized query content via WikiQueryInput.query field | accept | Query strings are user-supplied at the CLI/MCP boundary; no upper bound enforced in Phase 3 (typical queries < 200 chars). If runaway query strings become a problem, add `Field(max_length=2000)` in a future phase. |
</threat_model>
