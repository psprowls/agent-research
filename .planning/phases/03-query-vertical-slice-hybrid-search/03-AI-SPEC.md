## 1. System Classification

**System Type:** Hybrid (RAG + Multi-Agent)

**Description:**
A developer-facing knowledge base Q&A system that answers natural-language questions about a codebase by searching a local Markdown wiki vault (structured with YAML frontmatter, wikilinks, and layout blocks), extracting relevant passages in parallel via lightweight "librarian" model calls (Haiku), and synthesizing a final answer with `[[wikilink]]` citations via a stronger "synthesizer" model call (Sonnet). The system is a cost-optimized replacement for an existing Claude Code plugin (lattice-wiki): it must deliver answers of equivalent depth and citation structure while running entirely on AWS Bedrock at significantly lower cost. The primary users are the vault author (a solo developer) who already knows the codebase and will immediately recognize shallow, stale, or fabricated answers.

**Critical Failure Modes:**
1. **Fabricated code path or module reference** — the answer cites a file path, function name, or module that does not exist in the vault or codebase. The developer follows the reference, finds nothing, and loses trust in the tool.
2. **Correct answer, wrong vault page cited** — the `[[wikilink]]` in the answer points to a page that is real but does not contain the information the answer draws from. The developer follows the citation to verify and finds it unsupported.
3. **Answer drawn from stale vault content without disclosure** — the vault page the system found was last updated months ago and describes a since-removed API or pattern. The system presents the outdated information as current fact with no staleness signal.
4. **Top-K search misses the only relevant page** — the hybrid search ranks an irrelevant page above the one page that contains the answer. The synthesizer then answers "no information found" or produces a tangential answer. The developer concludes the wiki is missing the information when it is not.
5. **Answer confident about ambiguous wiki content** — the vault has two pages with conflicting descriptions of the same component (a known wiki maintenance problem). The system synthesizes a single confident answer that silently picks one version without noting the contradiction.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Internal developer tooling — codebase knowledge management and documentation Q&A

**User Population:** Solo developer (Pat) who is the vault author, with open-source distribution planned; future users are developers maintaining their own lattice-wiki vaults. All users have deep context about their own codebase and will immediately recognize incorrect file paths, wrong module names, or outdated architectural descriptions.

**Stakes Level:** Medium — answers are not life-critical, but incorrect answers waste developer time (following dead references, trusting stale architecture descriptions) and erode confidence in the tool. The primary cost of failure is time loss and tool abandonment, not safety incidents.

**Output Consequence:** The developer reads the answer and takes action: navigates to a cited file, runs a command described in the answer, or makes an architectural decision based on the synthesized explanation. A wrong `[[wikilink]]` citation sends the developer to the wrong page; a fabricated code path sends them searching for something that does not exist. Stale answers can propagate incorrect assumptions into new code.

### What Domain Experts Evaluate Against

**Dimension:** Citation traceability
Good (domain expert would accept): Every factual claim in the answer is backed by a `[[wikilink]]` citation to a real vault page that visibly contains that information when opened.
Bad (domain expert would flag): The answer includes `[[wikilinks]]` but the cited pages do not contain the information the answer claims they do — or the answer makes factual claims with no citation at all.
Stakes: Critical
Source: Practitioner standard — a developer verifying an answer opens every citation; an unsupported citation is indistinguishable from a fabrication.

---

**Dimension:** Code path and module name accuracy
Good (domain expert would accept): File paths, function names, class names, and module references in the answer exactly match what exists in the codebase (e.g., `cores/subagent-runtime/src/subagent_runtime/pool.py`).
Bad (domain expert would flag): The answer refers to `pool/subagent.py` or `SubAgentPool.dispatch()` — plausible-sounding paths or method names that do not exist. Developer opens the terminal, finds nothing, and reports a hallucination.
Stakes: Critical
Source: Amazon Science research on code LLM hallucinations (2024): API hallucinations — including non-existent method names and file paths — constitute up to 15% of code LLM errors and occur even with in-context documentation.

---

**Dimension:** Answer depth vs. the existing tool's baseline
Good (domain expert would accept): The answer covers the same conceptual scope as the current lattice-wiki plugin would: explains the relevant component, names the files involved, describes the interface, and notes any known caveats documented in the vault.
Bad (domain expert would flag): The answer is one or two sentences where lattice-wiki would have given a paragraph — or the answer omits well-documented caveats that the vault explicitly records (e.g., "this API is deprecated, use X instead").
Stakes: High
Source: Project requirement — the core value proposition is "output comparable in depth and structure to the current lattice-wiki result" (ROADMAP.md Phase 3, success criterion 1).

---

**Dimension:** Staleness transparency
Good (domain expert would accept): When the retrieved vault page has a `last_updated` frontmatter date that is significantly older than the vault's most recent activity, the answer notes this ("per the wiki page last updated in March, which may not reflect recent changes").
Bad (domain expert would flag): The answer presents information from a vault page that the developer knows to be outdated without any hedge — leading them to act on superseded information.
Stakes: High
Source: Developer knowledge base practitioner knowledge — wiki pages go stale; a tool that does not signal staleness can be more harmful than no tool.

---

**Dimension:** Search recall — "the answer is in the vault"
Good (domain expert would accept): For a query whose answer is documented in exactly one vault page, that page appears in the top-K search results and its content is reflected in the answer.
Bad (domain expert would flag): The synthesizer says "I could not find information about X" when the developer knows there is a vault page titled exactly "X" — meaning the hybrid search failed to surface the correct page.
Stakes: High
Source: RAG evaluation practice (2025): retrieval recall is the most common root cause of RAG answer gaps; citation accuracy in RAG systems without explicit attribution training averages 65–70% (Allen Institute for AI).

### Known Failure Modes in This Domain

1. **Low-frequency terminology zero-recall** — queries using terms introduced recently (new package names, renamed modules) may not appear in the frozen BM25 vocabulary; the tokenizer silently drops these tokens, causing BM25 to return nothing for what should be an easy exact-match query. The embedding half may partially recover, but if both signals miss, the page will not be retrieved.

2. **Cross-page contradiction synthesis** — the vault accumulates contradictions over time (old ADR vs. new ADR, refactored vs. unrefactored module docs). The synthesizer receives excerpts from both and produces a confident merged answer that is internally inconsistent but sounds coherent. The developer cannot tell which version is current without reading both pages manually.

3. **Correct page, wrong excerpt** — the librarian's extraction step surfaces a tangential passage from a page that contains the answer elsewhere. The synthesizer answers from the extracted excerpt rather than the key passage, producing a technically grounded but misleading answer. This is specific to the two-stage librarian/synthesizer design.

4. **Embedding index drift after bulk vault edits** — if many pages are edited between queries (e.g., after a scan command updates all package stubs), the embedding index falls out of sync until the incremental rebuild runs. Queries in this window may match stale embeddings, surfacing old content while the BM25 index (rebuilt on load) surfaces new content. RRF fusion over mismatched indices degrades ranking quality silently.

### Regulatory / Compliance Context

None identified for this deployment context. The system is internal developer tooling operating on a private, local Markdown vault with no PII, no regulated data, no end-user exposure, and no financial or medical domain content. No HIPAA, GDPR, FCA, or SOC 2 obligations apply to this specific system's outputs.

If the vault is ever used to store customer data, compliance context must be reassessed.

## 2. Framework Decision

**Selected Framework:** deepagents 0.6.1 + LangChain 1.3.0 + LangGraph 1.2.0 + langchain-aws 1.4.6

**Version:** deepagents==0.6.1, langchain==1.3.0, langgraph==1.2.0, langchain-aws==1.4.6

**Rationale:**
The stack is already in place and the Phase 3 pattern — parallel librarian fan-out via SubagentPool followed by a single synthesizer call — maps directly onto deepagents' built-in primitives with no new framework surface area. BedrockEmbeddings (Titan v2, 1536-dim) and bm25s 0.3.8 handle the hybrid search layer; RRF fusion is pure Python. Both delivery surfaces (FastMCP wiki_query tool and Typer CLI subcommand) share the single `commands/query.py` implementation, consistent with patterns established in Phases 1 and 2. AWS Bedrock is the sole provider; no direct Anthropic API exposure.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph directly (without deepagents) | deepagents IS LangGraph with SubagentPool, filesystem tools, and planning middleware already wired; dropping to bare LangGraph would require rebuilding those primitives from scratch with no benefit for a solo-developer, single-provider project |
| LlamaIndex | RAG-first framework with no deepagents integration; adding it would create a cross-framework seam in a stack already committed to LangChain/deepagents |
| CrewAI | No langchain-aws / ChatBedrockConverse binding; doesn't integrate with the existing SubagentPool and FanOutResult types |
| OpenAI Agents SDK | OpenAI-optimized; incompatible with Bedrock-only constraint |

**Vendor Lock-In Accepted:** Partial — AWS Bedrock only (no direct Anthropic API). All framework libraries are open-source and model-agnostic; Bedrock lock-in is intentional and documented as a v1 constraint.

### Entry Point Pattern

The query pipeline has two stages: (1) hybrid search to select candidate pages, then
(2) parallel librarian fan-out followed by a single synthesizer call. Both the CLI
subcommand and the MCP tool call `run_query()` from `commands/query.py`.

```python
# agents/code-wiki-agent/src/code_wiki_agent/commands/query.py
from __future__ import annotations

import re
import sqlite3
import struct
from dataclasses import dataclass
from pathlib import Path

import bm25s
from bm25s.tokenization import Tokenizer
from langchain_aws import BedrockEmbeddings
from langchain_core.messages import HumanMessage, SystemMessage

from model_adapter.loader import make_llm, load_role_config
from subagent_runtime.pool import SubagentPool, FanOutResult
from vault_io._workspace import resolve_wiki_and_repo


# ---- Token filter matching lattice-wiki-core reference behavior ----
_TOKEN_RE = re.compile(r"[a-zA-Z0-9][a-zA-Z0-9_\-']+")
_STOPWORDS = {
    "the", "a", "an", "and", "or", "but", "if", "to", "of", "in", "on",
    "at", "for", "by", "with", "from", "is", "are", "was", "were", "be",
    "been", "this", "that", "it", "as", "we", "you", "they", "not", "no",
    "do", "does", "did", "will", "would", "can", "could", "should", "also",
}

LIBRARIAN_SYSTEM = (
    "You are a wiki librarian. Given a user query and a single wiki page, "
    "extract every passage from the page that is directly relevant to the query. "
    "Quote the relevant text verbatim, annotating each excerpt with its wikilink. "
    "If the page contains nothing relevant, respond with exactly: NO_RELEVANT_CONTENT"
)

SYNTHESIZER_SYSTEM = (
    "You are a wiki synthesizer. Given a user query and a set of excerpts from "
    "relevant wiki pages, produce a concise, accurate answer. "
    "Cite every wiki page you draw from using [[wikilink]] notation. "
    "Include direct file paths and code references when present in the excerpts."
)


@dataclass
class QueryResult:
    answer: str
    citations: list[str]        # [[wikilink]] strings found in the answer
    pages_drilled: int
    search_scores: dict         # {page_path: {"bm25": float, "embed": float, "rrf": float}}


def _build_tokenizer() -> Tokenizer:
    """Build the bm25s Tokenizer that replicates lattice-wiki-core's TOKEN_RE behavior."""
    return Tokenizer(
        lower=True,
        splitter=r"[a-zA-Z0-9][a-zA-Z0-9_\-']+",  # same regex as lattice TOKEN_RE
        stopwords=list(_STOPWORDS),
    )


def _rrf_fuse(
    bm25_ranks: dict[str, int],
    embed_ranks: dict[str, int],
    k: int = 60,
) -> dict[str, float]:
    """Reciprocal Rank Fusion. score = 1/(k+rank_bm25) + 1/(k+rank_embed)."""
    all_pages = set(bm25_ranks) | set(embed_ranks)
    n = len(all_pages)
    return {
        p: 1.0 / (k + bm25_ranks.get(p, n + k)) + 1.0 / (k + embed_ranks.get(p, n + k))
        for p in all_pages
    }


def _cosine_search_sqlite(
    vault_path: Path,
    query_vec: list[float],
    top_k: int,
) -> list[tuple[str, float]]:
    """Linear cosine scan over embeddings in search.db. Returns (path, score) pairs."""
    db_path = vault_path / ".code-wiki" / "search.db"
    conn = sqlite3.connect(str(db_path))
    try:
        rows = conn.execute("SELECT path, embedding FROM pages").fetchall()
    finally:
        conn.close()

    import math
    q_mag = math.sqrt(sum(x * x for x in query_vec))
    results = []
    for path, blob in rows:
        vec = struct.unpack(f"{len(blob) // 4}f", blob)
        dot = sum(a * b for a, b in zip(query_vec, vec))
        v_mag = math.sqrt(sum(x * x for x in vec))
        score = dot / (q_mag * v_mag) if (q_mag and v_mag) else 0.0
        results.append((path, score))
    results.sort(key=lambda x: x[1], reverse=True)
    return results[:top_k]


def _extract_wikilinks(text: str) -> list[str]:
    return re.findall(r"\[\[([^\]]+)\]\]", text)


async def run_query(
    query: str,
    vault_path: Path,
    top_k: int = 5,
) -> QueryResult:
    """End-to-end query: hybrid search -> librarian fan-out -> synthesis."""
    wiki, _ = resolve_wiki_and_repo(vault_path)

    # 1. BM25 search (synchronous — bm25s has no async API)
    bm25_index_dir = vault_path / ".code-wiki" / "bm25"
    retriever = bm25s.BM25.load(str(bm25_index_dir), load_corpus=True)
    tokenizer = _build_tokenizer()
    # update_vocab=False: query must not expand the frozen index vocabulary
    query_tokens = tokenizer.tokenize([query], update_vocab=False, return_as="ids")
    bm25_results, bm25_raw_scores = retriever.retrieve(query_tokens, k=top_k * 3)

    # 2. Embedding search (one embed_query call — no per-document loop here)
    embeddings = BedrockEmbeddings(
        model_id="amazon.titan-embed-text-v2:0",   # no "us." prefix for Titan
        region_name="us-east-1",
        # normalize=True applies L2 normalization; cosine sim then equals dot product
        normalize=True,
    )
    query_vec = embeddings.embed_query(query)  # sync; runs boto3 in thread internally
    embed_hits = _cosine_search_sqlite(vault_path, query_vec, top_k=top_k * 3)

    # 3. RRF fusion
    bm25_rank_map = {str(bm25_results[0, i]): i + 1 for i in range(bm25_results.shape[1])}
    embed_rank_map = {path: i + 1 for i, (path, _) in enumerate(embed_hits)}
    fused = _rrf_fuse(bm25_rank_map, embed_rank_map)
    top_pages = sorted(fused, key=fused.get, reverse=True)[:top_k]

    # Build lookup maps for score reporting
    bm25_score_map = {str(bm25_results[0, i]): float(bm25_raw_scores[0, i])
                      for i in range(bm25_results.shape[1])}
    embed_score_map = dict(embed_hits)

    # 4. Librarian fan-out (parallel, max 5 concurrent — from models.toml)
    librarian_llm = make_llm("librarian")
    role_cfg = load_role_config("librarian")
    pool = SubagentPool(trace_dir=vault_path / ".code-wiki" / "traces")

    async def drill_page(page_path: str) -> str:
        page_text = (wiki / page_path).read_text()
        msgs = [
            SystemMessage(content=LIBRARIAN_SYSTEM),
            HumanMessage(content=f"Query: {query}\n\nPage ({page_path}):\n{page_text}"),
        ]
        resp = await librarian_llm.ainvoke(msgs)
        return resp.content

    fan_result: FanOutResult = await pool.run_all(
        items=top_pages,
        task=drill_page,
        role="librarian",
        model_id=role_cfg["model_id"],
        max_concurrency=role_cfg["max_concurrency"],  # 5 from models.toml
    )

    # 5. Synthesizer (single call — Sonnet, max_tokens=4096 from models.toml)
    excerpts = "\n\n---\n\n".join(
        f"[{item}]\n{result}"
        for item, result in fan_result.successes
        if result.strip() != "NO_RELEVANT_CONTENT"
    )
    synth_llm = make_llm("synthesizer")
    synth_msgs = [
        SystemMessage(content=SYNTHESIZER_SYSTEM),
        HumanMessage(content=f"Query: {query}\n\nLibrarian excerpts:\n{excerpts}"),
    ]
    synth_resp = await synth_llm.ainvoke(synth_msgs)

    return QueryResult(
        answer=synth_resp.content,
        citations=_extract_wikilinks(synth_resp.content),
        pages_drilled=len(fan_result.successes),
        search_scores={
            p: {
                "bm25": bm25_score_map.get(p, 0.0),
                "embed": embed_score_map.get(p, 0.0),
                "rrf": fused.get(p, 0.0),
            }
            for p in top_pages
        },
    )
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `SubagentPool.run_all()` | Async fan-out: dispatches N pages to the librarian role in parallel with a semaphore cap, returns `FanOutResult` with partial-failure isolation via `asyncio.gather(return_exceptions=True)` | Phase 3 librarian fan-out — pass `top_k` page paths as `items`, `drill_page` closure as `task` |
| `BedrockEmbeddings` | `langchain-aws` class wrapping boto3 for Titan v2 vector generation; `embed_query()` and `embed_documents()` are both synchronous (boto3 under the hood); `aembed_documents()` is pseudo-async via `run_in_executor` | Building the embedding index on ingest; single `embed_query()` call at search time |
| `bm25s.BM25` + `Tokenizer` | Pure-Python BM25 index; `index()` builds sparse matrix; `save(dir, corpus=corpus)` persists to `.code-wiki/bm25/`; `load(dir, load_corpus=True)` restores; `Tokenizer` handles vocab building and persistence | BM25 half of hybrid search; tokenizer is built once, frozen after `index()`, persisted via `tokenizer.save_vocab()` |
| RRF fusion | Pure-Python `1/(k+rank_bm25) + 1/(k+rank_embed)`, `k=60`; no alpha parameter to tune; produces a single ranked list | Merging BM25 and embedding results into the final `top_k` candidate page list |
| `FastMCP` + `Context` | MCP tool registration; `Context` parameter injected by FastMCP gives the tool `await ctx.report_progress(progress, total, message)` for notifying the client during long-running fan-out | `wiki_query` MCP tool — emit one progress notification per completed librarian call |

### Common Pitfalls

1. **`BedrockEmbeddings.embed_documents()` calls boto3 once per document, not in a batch.** The `_iteratively_embed_documents()` implementation loops and calls `_embedding_func(text)` for each item individually. At 50-page vaults this is tolerable; at 500+ pages, the sequential Bedrock round-trips dominate latency. Mitigation: use a thread pool (`asyncio.to_thread` or `loop.run_in_executor`) with a capped concurrency to parallelize embedding calls during index build, not during query time.

2. **`BedrockEmbeddings.aembed_documents()` is pseudo-async — it does not release the GIL or issue parallel HTTP requests.** It wraps the sync boto3 client via `run_in_executor`. If you `await embeddings.aembed_documents(all_pages)` you get sequential execution in a thread pool, not true concurrency. For true parallel embedding, call `embed_query()` via `asyncio.gather` with individual text items and a semaphore.

3. **bm25s `Tokenizer` vocabulary is frozen after `index()`.** Calling `tokenizer.tokenize(query, update_vocab=False)` at query time silently drops tokens not seen during indexing. Always persist the tokenizer alongside the BM25 index using `tokenizer.save_vocab(bm25_dir)` and `tokenizer.save_stopwords(bm25_dir)`, then reload with `new_tokenizer.load_vocab(bm25_dir)` before querying. Forgetting this causes silent zero-recall for any new terminology used in a query.

4. **`SubagentPool` semaphore must be created inside `run_all()`, not in `__init__` or at module level.** The semaphore is bound to the event loop running at creation time. pytest-asyncio spins a new loop per test; a semaphore created in a different loop raises `RuntimeError: Future attached to a different loop`. The existing `pool.py` already does this correctly — do not move the `asyncio.Semaphore(max_concurrency)` call out of `run_all()`.

5. **Titan v2 embedding model ID has no cross-region `us.*` prefix.** The correct `model_id` for `BedrockEmbeddings` is `"amazon.titan-embed-text-v2:0"`, not `"us.amazon.titan-embed-text-v2:0"`. Cross-region inference profiles (`us.*`) apply to Claude chat models. Using the prefixed ARN for Titan will return an `AccessDeniedException` even if the model is enabled. Verify in the AWS Bedrock console under "Model access" in your target region.

6. **MCP `ctx.report_progress()` is a no-op if the client did not send a `progressToken`.** FastMCP's `Context.report_progress(progress, total, message)` silently does nothing when no token was provided. Claude Code and DeepAgents CLI send `progressToken` by default; direct `mcp` SDK test clients do not unless explicitly configured. Do not use progress notifications as a synchronization or signaling mechanism — they are informational only.

**Model Configuration:**

All models are resolved via `make_llm(role)` from `cores/model-adapter`. Do not hardcode model IDs in `commands/query.py`. The Phase 3 roles from `models.toml` are:

| Role | Model ID | max_tokens | max_concurrency | Purpose |
|------|----------|-----------|-----------------|---------|
| `librarian` | `us.anthropic.claude-haiku-4-5-20251001-v1:0` | 2048 | 5 | Page drilling — extracts relevant excerpts per page |
| `synthesizer` | `us.anthropic.claude-sonnet-4-6` | 4096 | 3 | Final answer with `[[wikilink]]` citations |

For embeddings, `BedrockEmbeddings` is instantiated directly (not via `make_llm`) with:

```python
BedrockEmbeddings(
    model_id="amazon.titan-embed-text-v2:0",  # no "us." prefix
    region_name="us-east-1",                  # must match the region in models.toml
    normalize=True,                            # L2 normalize; cosine sim = dot product
    # dimensions param omitted: use default 1024 (not 1536 as earlier stated;
    # AWS docs confirm 1024 is the default output size for Titan v2)
)
```

Note: AWS docs state Titan v2 default output is 1024 dimensions (not 1536). The SQLite
`embedding BLOB` stores `struct.pack(f"{len(vec)}f", *vec)` — 4 bytes per float, so
1024-dim vectors = 4096 bytes per page row.

**Core Pattern:**

The query command follows a strict linear pipeline: hybrid search (sync BM25 + sync
embedding query) -> RRF fusion (pure Python) -> async librarian fan-out (SubagentPool)
-> async synthesizer call. Only the fan-out and synthesizer are async; the search phase
runs synchronously and completes before the event loop is entered.

For the MCP tool, wrap `run_query()` in an async function decorated with `@mcp.tool`
that accepts a `ctx: Context` parameter for progress reporting:

```python
# agents/code-wiki-agent/src/code_wiki_mcp/server.py (addition to existing file)
from mcp.server.fastmcp import FastMCP, Context
from pydantic import BaseModel

from code_wiki_agent.commands.query import run_query, QueryResult
from vault_io._workspace import resolve_wiki_and_repo


class WikiQueryInput(BaseModel):
    query: str
    vault_path: str = ""       # empty -> resolve from workspace
    top_k: int = 5             # 3-10 range; 5 is the default (D-07)


class WikiQueryOutput(BaseModel):
    answer: str
    citations: list[str]
    pages_drilled: int
    search_scores: dict        # {page_path: {"bm25": float, "embed": float, "rrf": float}}


@mcp.tool(
    name="wiki_query",
    description="Query the code wiki using hybrid BM25+embedding search with parallel librarian analysis.",
)
async def wiki_query(input: WikiQueryInput, ctx: Context) -> WikiQueryOutput:
    from pathlib import Path
    vault = Path(input.vault_path) if input.vault_path else Path.cwd()

    # Emit initial progress so the client knows work has started
    await ctx.report_progress(progress=0, total=input.top_k, message="Starting hybrid search")

    result: QueryResult = await run_query(
        query=input.query,
        vault_path=vault,
        top_k=input.top_k,
    )

    # Final progress — fan-out complete
    await ctx.report_progress(
        progress=result.pages_drilled,
        total=input.top_k,
        message=f"Synthesized from {result.pages_drilled} pages",
    )

    return WikiQueryOutput(
        answer=result.answer,
        citations=result.citations,
        pages_drilled=result.pages_drilled,
        search_scores=result.search_scores,
    )
```

**Tool Use:**

The `wiki_query` MCP tool is a thin wrapper. It does no business logic — it validates
input via Pydantic, delegates to `run_query()`, and serializes the `QueryResult`
dataclass to `WikiQueryOutput`. The `ctx.report_progress()` calls bracket the work.

The librarian and synthesizer LLMs do not use LangChain tool calling in Phase 3 —
they receive structured prompts and return plain text. Tool calling (for functions like
"read file from vault") is deferred to future phases; librarians in Phase 3 receive the
full page content in the prompt.

**State Management:**

Phase 3 has two persistent state stores, both in `.code-wiki/`:

- **`bm25/`** — bm25s manages its own joblib-based directory serialization. The index
  is rebuilt on demand (sha256 change detection). At query time, `BM25.load()` is called
  at the start of `run_query()`; the loaded retriever is not cached between calls in v1
  (vault is small; load is fast). If load latency becomes a concern in v2, cache the
  retriever as a module-level singleton guarded by a threading lock.

- **`search.db`** — SQLite in WAL mode. Schema:
  ```sql
  CREATE TABLE IF NOT EXISTS pages (
      path         TEXT PRIMARY KEY,
      content_hash TEXT NOT NULL,
      embedding    BLOB NOT NULL
  );
  PRAGMA journal_mode=WAL;
  ```
  Upsert pattern: `INSERT OR REPLACE INTO pages (path, content_hash, embedding) VALUES (?, ?, ?)`.
  WAL mode is safe for concurrent readers during query while an index rebuild writes.
  `search.db` is opened and closed within each `_cosine_search_sqlite()` call — no
  persistent connection at module level (avoids threading issues under pytest).

`run_query()` is stateless across calls — it opens the BM25 index and SQLite DB, does
its work, and closes them. No in-memory caching of embeddings or BM25 retriever between
requests in v1.

**Context Window Strategy:**

- **Librarian calls (Haiku, 2048 tokens out):** Each librarian call receives the
  original query + one full page of markdown. Vault pages are typically 200-1500 tokens.
  If a page exceeds ~6000 tokens (leaving room for system prompt and query), truncate to
  the first 6000 tokens before sending; log a warning. This is uncommon in a code wiki
  but protects against runaway pages.

- **Synthesizer call (Sonnet, 4096 tokens out):** The synthesizer receives all librarian
  excerpts concatenated. Excerpts are partial (each librarian is extracting, not
  reproducing the full page), so typical synthesizer input is 500-3000 tokens across
  5 pages. If `top_k=10` and librarians are verbose, the input can reach 6000-8000
  tokens; this is within Sonnet's context window. If the concatenated excerpt string
  exceeds 15000 tokens (a safety threshold), truncate per-excerpt to 1500 tokens before
  concatenation and log a warning.

- **No RAG chunking in Phase 3:** Pages are passed whole to librarians. The librarian
  is responsible for extracting the relevant portion. This avoids the chunk-boundary
  problem where key context spans a chunk split.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs with Pydantic

The `QueryResult` dataclass is the single canonical return type for `run_query()`. The
MCP tool output (`WikiQueryOutput`) is a Pydantic `BaseModel` that mirrors its fields.
Neither the librarian nor the synthesizer is asked to produce structured JSON in Phase 3
— they produce natural language, and structured extraction happens in Python code
(`_extract_wikilinks()` for citations, `dataclasses.asdict()` for serialization).

This is intentional: asking Haiku (librarian) to produce structured JSON excerpts adds
prompt complexity and a JSON parse failure mode. Plain text extraction is more robust at
2048 tokens output limit.

For the synthesizer, the answer is validated in Python, not by asking the LLM to
produce JSON:

```python
# commands/query.py — structured output from synthesizer (no LLM retry needed)
from __future__ import annotations

import re
from dataclasses import dataclass


@dataclass
class QueryResult:
    answer: str
    citations: list[str]       # extracted by _extract_wikilinks(), not by LLM
    pages_drilled: int
    search_scores: dict        # built from search layer, not from LLM


def _extract_wikilinks(text: str) -> list[str]:
    """Extract [[target]] citation strings from synthesizer output."""
    return re.findall(r"\[\[([^\]]+)\]\]", text)


# If you ever need the synthesizer to produce structured JSON (e.g., a confidence
# score), use ChatBedrockConverse.with_structured_output() with a Pydantic model:
from pydantic import BaseModel

class SynthesizerOutput(BaseModel):
    answer: str
    confidence: float   # 0.0-1.0

# Then:
# structured_llm = make_llm("synthesizer").with_structured_output(SynthesizerOutput)
# result = await structured_llm.ainvoke(msgs)
# result is a SynthesizerOutput instance — no JSON parsing needed.
#
# Retry logic for structured output failures:
# ChatBedrockConverse.with_structured_output() raises OutputParserException on bad JSON.
# Wrap in a retry loop: max 2 retries, log the raw LLM output on each failure,
# raise after 2 retries with the original OutputParserException for the caller to handle.
```

If structured output is added in a future phase:
- Use `llm.with_structured_output(PydanticModel)` from LangChain — this is the
  correct integration point for ChatBedrockConverse.
- Retry budget: 2 retries maximum. Log the raw LLM response text on each retry.
  After 2 retries, surface the `OutputParserException` to the caller — do not silently
  return a default value, as that masks model degradation.
- Do not use `instructor` — it targets the OpenAI SDK directly and does not integrate
  with `langchain-aws`.

### 4b.2 Async-First Design

`run_query()` is an `async def` function. The two async operations are:
(a) `pool.run_all()` — concurrent librarian fan-out via `asyncio.gather`
(b) `synth_llm.ainvoke()` — single synthesizer call

The BM25 and embedding search steps are synchronous (bm25s is sync-only; `embed_query()`
wraps sync boto3). They run before the event loop is entered for the LLM calls and add
no async overhead.

**The one mistake to avoid:** `asyncio.run()` inside an already-running event loop.

```python
# WRONG — crashes with "This event loop is already running" in any async context
# (MCP tool handlers, pytest-asyncio tests, FastAPI routes):
def wiki_query_sync(input):
    result = asyncio.run(run_query(...))   # RuntimeError if event loop is active

# CORRECT in CLI context — Typer commands are sync; asyncio.run() is valid:
@app.command()
def query(query_text: str, ...):
    result = asyncio.run(run_query(query_text, vault_path, top_k=top_k))

# CORRECT in MCP tool context — MCP tool handlers are async:
@mcp.tool(name="wiki_query")
async def wiki_query(input: WikiQueryInput, ctx: Context) -> WikiQueryOutput:
    result = await run_query(input.query, vault_path, input.top_k)
    return WikiQueryOutput(...)

# CORRECT in tests — use pytest-asyncio:
# @pytest.mark.asyncio
# async def test_run_query():
#     result = await run_query(...)
```

**Stream vs await:** In Phase 3, use `await llm.ainvoke(msgs)` (not streaming) for both
librarian and synthesizer calls. Structured result validation (`_extract_wikilinks`) and
`QueryResult` construction require the complete response. Streaming is only useful when
you want to forward tokens to a user UI in real time — the MCP tool returns a complete
`WikiQueryOutput`, so streaming would require buffering the entire response anyway.
If a future streaming CLI mode is added, stream the synthesizer only; never stream
librarian calls (their output is intermediate, not user-facing).

### 4b.3 Prompt Engineering Discipline

**System vs user prompt separation:** Always use `SystemMessage` for role definition
and constraints, `HumanMessage` for the dynamic content (query + page text). Never
embed role instructions in the `HumanMessage` — this confuses Claude's instruction
hierarchy and degrades output quality.

```python
# CORRECT — role in system, content in human:
msgs = [
    SystemMessage(content=LIBRARIAN_SYSTEM),        # static; defined as module constant
    HumanMessage(content=f"Query: {query}\n\nPage ({page_path}):\n{page_text}"),
]

# WRONG — mixing role and content in a single message:
msgs = [HumanMessage(content=f"You are a librarian. Query: {query} Page: {page_text}")]
```

**Constants, not f-strings for system prompts:** `LIBRARIAN_SYSTEM` and
`SYNTHESIZER_SYSTEM` are module-level string constants. This makes prompt versions
diffable in git, testable in unit tests without LLM calls, and avoids accidental
runtime mutation.

**Few-shot examples:** In Phase 3, librarians and synthesizer do not need few-shot
examples — the tasks are well-defined (extract relevant text; produce cited answer).
If output quality degrades in eval, add 1-2 inline few-shot examples to `LIBRARIAN_SYSTEM`
before reaching for dynamic retrieval.

**max_tokens is set explicitly:** Both roles have `max_tokens` set in `models.toml`
(librarian: 2048, synthesizer: 4096). `make_llm(role)` passes this to
`ChatBedrockConverse(max_tokens=...)`. Never leave `max_tokens` unset in production —
an unbounded generation on Sonnet at $3/MTok output tokens can produce 10x the expected
cost on a runaway response.

**The `NO_RELEVANT_CONTENT` sentinel:** `LIBRARIAN_SYSTEM` instructs librarians to
respond with exactly `NO_RELEVANT_CONTENT` when the page has nothing relevant. Filter
these out before passing excerpts to the synthesizer. This prevents the synthesizer from
receiving noise that could dilute the answer or generate hallucinated citations.

### 4b.4 Context Window Management

**Per librarian call (Haiku, 2048 tokens output, ~200K token context):**
- Input = system prompt (~120 tokens) + query (~30 tokens) + page text (200-1500 tokens
  typical for a code wiki page).
- Haiku's context window is large; typical page inputs are well within budget.
- Guard: if `len(page_text) > 24000 characters` (~6000 tokens), truncate to 24000
  characters and append `\n[TRUNCATED]`. Log a WARNING with the page path.
- Do not split the page into chunks and send multiple librarian calls — one librarian
  per page is the Phase 3 contract. Chunk-splitting is a Phase 5+ optimization.

**Synthesizer call (Sonnet, 4096 tokens output):**
- Input = system prompt (~150 tokens) + query (~30 tokens) + all non-null excerpts.
- Excerpts are partial (librarians extract relevant sections, not full pages), so
  typical input is 500-3000 tokens across 5 pages drilled.
- Guard: if `len(excerpts) > 60000 characters` (~15000 tokens), truncate each excerpt
  to `min(len(excerpt), 6000 characters)` before concatenation. This is a last-resort
  safety valve; it should not trigger in normal vault operation.
- The synthesizer does not receive raw page content — only librarian excerpts. This
  two-stage design is the primary context management strategy: librarians act as
  compression step before synthesis.

**BM25 + embedding candidate expansion:** The search layer retrieves `top_k * 3`
candidates from each backend before RRF fusion and final `top_k` selection. This
over-retrieval is intentional — RRF ranking on 15 candidates to select 5 is more
accurate than ranking on 5. The over-retrieved candidates never reach the LLM; they are
filtered in pure Python.

### 4b.5 Cost and Latency Budget

**Per query cost estimate (top_k=5, typical vault page ~600 tokens):**

| Step | Model | Input tokens | Output tokens | Cost (approx) |
|------|-------|-------------|---------------|---------------|
| embed_query (query) | Titan v2 | ~30 | — | $0.00002 |
| 5x librarian (parallel) | Haiku | 5 * ~800 = 4000 | 5 * ~400 = 2000 | $0.00135 |
| 1x synthesizer | Sonnet | ~2500 | ~600 | $0.01110 |
| **Total** | | | | **~$0.013** |

Pricing basis: Haiku $0.00025/K input + $0.00125/K output; Sonnet $0.003/K input +
$0.015/K output (on-demand Bedrock us-east-1, 2026-05-13 rates — verify current rates).
Titan v2: $0.00002/K tokens input.

**Latency estimate (top_k=5, 5 parallel librarians):**
- Hybrid search (sync): ~50-150ms (BM25 in-memory; SQLite linear scan for 100 pages)
- Librarian fan-out (5 parallel): ~800-1500ms (Haiku first-token latency ~300ms;
  2048 token output at ~100 tok/s = ~1200ms; dominates)
- Synthesizer (1 call): ~1500-3000ms (Sonnet at 4096 tokens out)
- **Total wall-clock: ~2.5-5s** for a typical query

**Caching strategy:**
- **Exact-match query cache:** Not implemented in Phase 3. If the same `(query, vault_path,
  top_k)` triple is issued twice, the full pipeline reruns. Add a simple in-memory LRU
  cache (`functools.lru_cache` does not work with `async def`; use a `dict` keyed on
  `hash((query, str(vault_path), top_k))` with a TTL or 100-entry cap) in Phase 5 if
  repeated queries become common.
- **Embedding cache:** `search.db` is the embedding cache — pages are only re-embedded
  when their sha256 content hash changes (D-02). The query vector is not cached (one
  `embed_query()` call per user query is negligible cost).
- **No semantic caching in Phase 3:** Semantic caching (finding a cached answer for a
  similar-but-not-identical query) requires a vector store of past queries and adds
  infrastructure complexity; deferred to v2.

**Cheaper model opportunities:**
- Librarian role already uses Haiku (the cheapest capable Claude model). No further
  downgrade is warranted for page drilling — Haiku 4.5 has sufficient instruction
  following for extraction tasks.
- If `top_k` is increased above 5, the librarian fan-out cost scales linearly with page
  count but the synthesizer cost stays roughly constant (excerpts are compressed by
  librarians). The cost-limiting lever is `top_k`, not model choice.
- A future "routing" step could classify the query complexity and use Sonnet only for
  complex multi-hop queries, falling back to Haiku as synthesizer for simple factual
  lookups. Not in Phase 3 scope.

- **Non-Claude Bedrock judge:** `us.amazon.nova-pro-v1:0` (preferred — same provider as

```bash
# Full eval (structural + LLM judges) — runs on demand or nightly,

  --eval-baseline=eval/baselines/lattice-wiki/ \

# Regression gate (EVAL-09) — fails CI if composite quality drops below threshold
# vs. the committed baseline for any role:
uv run --package eval-harness pytest -m "eval_regression" --regression-threshold=0.05
```

## 6. Guardrails

> Domain stakes are **Medium** (Section 1b): a single bad answer wastes developer time
> and erodes tool trust, but no safety / regulatory exposure exists. Therefore Phase 3
> uses **no hard-blocking online guardrails** — the vault author can recognize a bad
> answer on sight, so blocking degrades the tool more than it protects. Online guardrails
> are restricted to integrity checks (citation/schema validity) that can be repaired
> in-line without re-running the LLM pipeline. Everything else lives in the offline
> flywheel.

### Online (Real-Time)

These run on every `run_query()` invocation. Latency budget: the entire guardrail block
must add ≤ 50ms (the costly work — librarian + synthesizer calls — has already happened).

| # | Guardrail | Trigger | Intervention |
|---|-----------|---------|--------------|
| G1 | **Citation-resolution check** (D1) | Any `[[wikilink]]` in `QueryResult.answer` does not resolve to an existing vault page (case-insensitive lookup against `wiki/**/*.md`). | **Flag** — append a single-line warning to the answer: `[warning: N citation(s) did not resolve: <list>]`. Do **not** strip the answer; the developer needs to see what was attempted to recognize the failure mode. Increment the `citation_unresolved_total` counter logged to the JSONL trace for offline analysis. |
| G2 | **Output-schema validation** (D8) | `QueryResult` fails dataclass field shape, or MCP `WikiQueryOutput` fails Pydantic validation. | **Block** — raise a typed exception (`QueryOutputError`) before returning to the caller. The CLI maps this to a non-zero exit code with stderr message; the MCP tool returns a structured error response. This is the only blocking guardrail because a malformed `QueryResult` violates the wire contract with downstream consumers (DeepAgents CLI, scripted callers). |
| G3 | **Code-path sanity check** (D3) | The answer references a path matching `[a-z_][a-z0-9_/.]+\.(py\|md\|toml\|json)` that does **not** exist under the repo root and does **not** appear verbatim in any librarian excerpt in `fan_result.successes`. | **Flag** — append `[warning: unverified code path(s): <list>]` to the answer. Same rationale as G1: surface, do not suppress. |
| G4 | **Empty-result safety** (1b failure mode: top-K misses the only relevant page) | `fan_result.successes` is empty (all librarians returned `NO_RELEVANT_CONTENT` or failed), **and** the synthesizer still produced an answer with citations. | **Flag** — prepend `[warning: no librarian excerpts; answer is unsupported by retrieved pages]` and clear `QueryResult.citations` to `[]` to prevent G1 false-positives. The developer is then strongly signaled to re-query or check the vault directly. |

What is **explicitly not** an online guardrail (rationale: Medium stakes + solo developer):
- **Tone / safety profanity checks** — internal developer tool, single user, no
  user-facing reputation risk.
- **PII / data-leak filters** — vault contains no PII (Section 1b regulatory: none).
- **LLM-judge gating per request** — judge calls would double per-query cost and double
  latency for a flag the developer can self-detect.

### Offline (Flywheel)

These run on sampled traces, not in the hot path. Phase 3 writes the JSONL traces; Phase 4
adds the eval harness that consumes them. Sampling strategy in Section 7.

| # | Metric | Sampling Strategy | Action on Degradation |
|---|--------|-------------------|----------------------|
| F1 | **Citation grounding rate** (D2) | Weekly batch over all queries from the prior 7 days where `citations` is non-empty. Run the D2 LLM-judge panel on each citation. | Below 90% panel-agreed grounded → run model sweep (EVAL-04) on librarian role to identify whether a Haiku alternative improves extraction fidelity; consider tightening `LIBRARIAN_SYSTEM` to require verbatim quoting (already specified — degradation suggests the model is not following). |
| F2 | **Recall@5** (D5) | Whenever the reference dataset is rerun (release gates + monthly). | Below 0.9 → audit the failures: are the missed pages new content (BM25 vocab issue, 1b failure mode #1) or semantically distant (embedding gap)? Action: trigger BM25 reindex if vocabulary drift; revisit embedding model if semantic gap. |
| F3 | **Depth gap vs. baseline** (D4) | All eval-suite queries, every model-sweep run (EVAL-04, EVAL-07). | Mean depth score < 3.5 for any synthesizer candidate → that candidate is rejected from cost-frontier consideration; do not downgrade synthesizer below current Sonnet without passing this gate. |
| F4 | **Staleness hedge rate** (D6) | All `stale_page_expected`-tagged eval cases. | Hedge rate < 100% on tagged cases → modify `SYNTHESIZER_SYSTEM` to include explicit "if cited pages have old `last_updated`, note this" instruction; re-run eval. |
| F5 | **Contradiction detection rate** (D7) | All `contradiction_expected`-tagged eval cases. | < 100% surfacing on tagged cases → consider adding a `_detect_excerpt_conflict()` step in `run_query()` that flags conflicting excerpts to the synthesizer prompt; re-evaluate. Deferred until evidence of regression. |
| F6 | **Per-query cost drift** (D10) | All production queries (the population is small enough — solo developer — that sampling is full-coverage). | Median cost per query rising > 25% week-over-week → check `usage_metadata` for prompt bloat; check whether `top_k` defaults changed; check vault page size trends. |
| F7 | **Librarian sentinel emission rate** (D9) | All queries; count `NO_RELEVANT_CONTENT` responses in the JSONL traces. | Rate drops below 5% (suggesting librarians are inventing relevance) or rises above 60% (suggesting retrieval is feeding too many off-topic pages) → both warrant investigation, but the latter is more often a retrieval-recall problem than a librarian problem. |

Flywheel cadence: F1, F6, F7 weekly; F2–F5 on every model-sweep / release-gate run.
Pat reviews F1 and F2 reports manually (the only flywheel steps that have judge calls — a
human in the loop calibrates the judges per Section 1b expert role).

## 7. Production Monitoring

> "Production" here means Pat's daily local use against his own vault. There is no
> multi-tenant traffic, no hosted backend, no SLA. Monitoring stays local-only,
> file-based, and free of hosted-service dependencies, consistent with the Bedrock-only +
> open-source-ready posture of the project.

**Tracing Tool:** **JSONL traces written by `SubagentPool`** (OBS-01, Phase 2 deliverable)
to `<vault>/.code-wiki/traces/`, rendered by `code-wiki-agent trace <file>` (OBS-02).
Phase 3 inherits this for free — every librarian fan-out call emits a trace record.

No external tracing infrastructure is added in Phase 3. Specifically rejected:
- **LangSmith** — per-trace billing model is a non-starter for a personal-tool budget
  and violates the Bedrock-only / no-hosted-dependency posture.
- **Arize Phoenix** / **Langfuse** — would add a self-hosted service (Docker, port,
  retention policy) that adds maintenance overhead for one user. The JSONL files plus the
  existing `trace` renderer already give equivalent visibility for the access pattern
  Pat actually uses (look at one query in detail after the fact).

If a future phase needs multi-query aggregation (e.g., "find the slowest synthesizer call
this week"), the JSONL trace files are already structured for `jq` / DuckDB / Polars
analysis without instrumenting a tracing platform — keep that escape hatch open before
reaching for Phoenix.

**Trace record additions for Phase 3** (extends the Phase 2 schema; SubagentPool already
writes most of these):
- `role` — `"librarian"` or `"synthesizer"` (already present).
- `model_id` — exact Bedrock ARN used (already present; required for EVAL-08
  reproducibility once Phase 4 lands).
- `input_tokens` / `output_tokens` — from `usage_metadata` (already present).
- `cost_usd` — `null` in Phase 3, populated in Phase 4 (cost layer is deferred per
  Phase 3 boundary).
- `query_id` — new in Phase 3: a per-`run_query()` UUID so all librarian traces +
  synthesizer trace for one query can be joined offline. Wire this through
  `SubagentPool.run_all()` via a new optional `correlation_id` kwarg.
- `top_k`, `pages_retrieved`, `pages_drilled` — new query-level metadata fields
  written once per `run_query()` invocation as a `kind=query_summary` record alongside
  the per-subagent records.
- `search_scores` — per-page `{bm25, embed, rrf}` triple from `QueryResult` (SEARCH-06
  data; needed for D5 recall debugging).

**Key Metrics to Track** (computed from JSONL traces via the Phase 4 harness; no
additional instrumentation in Phase 3):

| Metric | Source | Why |
|--------|--------|-----|
| Median + p95 wall-clock per query | `query_summary` record `started_at`/`ended_at` | D10 latency gate |
| Median + p95 per-query cost | `query_summary` record sum of children `cost_usd` (Phase 4) | D10 cost gate; primary signal that motivates the whole project |
| Librarian success rate | `FanOutResult.successes` / `top_k` | Detects Bedrock throttling, page-read failures |
| `NO_RELEVANT_CONTENT` sentinel rate | librarian trace `output` text scan | F7 flywheel; the "librarian honesty" signal |
| Citation count per answer | `QueryResult.citations` length | D1/D2 input; trends below 1 mean answers becoming uncited |
| Unresolved-citation rate | G1 guardrail counter | Real-time integrity drift indicator |
| Search-score distribution | per-page `rrf` values in `query_summary` | Detects retrieval degradation (e.g., all RRF scores collapse to the floor → index corruption) |
| RRF-fusion lift | `rrf` rank vs. `bm25`-only and `embed`-only ranks for top result | Validates the hybrid approach is doing work vs. either single signal |

**Alert Thresholds** (Pat reviews on demand — no paging / push notification needed for
a personal tool; thresholds are documented so the Phase 4 harness can color-code reports):

| Signal | Yellow (investigate) | Red (act now) |
|--------|----------------------|---------------|
| Median wall-clock per query | > 6s | > 10s |
| Median cost per query (Phase 4) | > $0.020 | > $0.030 |
| Unresolved-citation rate (G1) | > 5% of queries in last 20 | > 15% of queries in last 20 |
| Librarian failure rate (per-page exceptions in `FanOutResult.errors`) | > 5% | > 20% |
| Recall@5 on reference dataset | < 0.90 | < 0.80 |
| Citation grounding (D2 judge) | < 0.90 | < 0.80 |
| Search-score top-1 RRF value | < 0.020 (suggests no strong match for any query) | < 0.010 sustained |

Red on cost or recall is a hard stop — the project's core value proposition is "cheaper
than lattice-wiki with comparable depth", so a regression on either invalidates the
deployment.

**Smart Sampling Strategy:**

For a solo-developer tool, traffic volume is low (single-digit queries per day in normal
use, dozens per day during active debugging). Sampling design:

- **Online**: **100% trace capture, always**. There is no volume justification to sample
  down — every query is a labeled example the eval flywheel can use. The JSONL files are
  small (low KB per query); a year of daily use is < 50 MB.
- **Offline (flywheel)**: a **prioritized review queue**, not random sampling — Pat's time
  is the bottleneck, not data volume.
  1. **Always review (manual)** — any query that triggered an online flag (G1, G3, G4).
     These are the highest-signal trace records: the system itself told us something
     looked off.
  2. **Always review (judge)** — every query with non-empty `citations` runs through D2
     citation-grounding judge weekly (F1).
  3. **Sampled review (judge)** — 20% random sample of the remaining queries each week
     for D4 depth, to catch slow drift before it shows up in flag rates.
  4. **Eval-dataset rerun** — full reference dataset (10→20 cases) on every model change
     and at minimum monthly, regardless of production volume.
- **Smart weighting** (`ai-evals.md` smart-sampling pattern): retries (none in Phase 3 — no
  retry layer), unusually short answers (< 50 chars), unusually long answers (> 4000
  chars), and queries where `pages_drilled < top_k * 0.5` (mass librarian failure) are
  all promoted into the manual review queue regardless of flag state.

The output of weekly review is a single appended row in `eval/reports/weekly.jsonl`
capturing counts per flag category, judge pass rate, and any newly observed failure mode —
this becomes the change log Pat scans before deciding whether to update prompts, models,
or the eval suite itself.
