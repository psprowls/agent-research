Module `graph_io.uri` (to be created):

```
@dataclass(frozen=True)
class RepoContext:
    org: str
    repo: str

def repo_uri(ctx: RepoContext) -> str: ...               # "repo:{org}/{repo}"
def pkg_uri(ctx: RepoContext, name: str) -> str: ...     # "pkg:{org}/{repo}/{name}"
def subpkg_uri(ctx: RepoContext, pkg_name: str, dotted_path: str) -> str: ...
                                                          # "subpkg:{org}/{repo}/{pkg_name}/{dotted.path}"
def file_uri(ctx: RepoContext, rel_path: str) -> str: ...
                                                          # "file:{org}/{repo}/{rel/path/to/file.py}"
def entry_point_uri(ctx: RepoContext, pkg_name: str, ep_name: str) -> str: ...
                                                          # "entry_point:{org}/{repo}/{pkg_name}/{ep_name}"
def test_suite_uri(ctx: RepoContext, suite_name: str) -> str: ...
                                                          # "test_suite:{org}/{repo}/{suite_name}"
def domain_uri(name: str) -> str: ...                    # "domain:{name}"  -- NO ctx (v1.6)
def parse_remote_url(url: str) -> tuple[str, str] | None: ...
```

Suggested (not locked) regexes from CONTEXT.md <specifics>:
- SSH:   `^git@[^:]+:([^/]+)/(.+?)(?:\.git)?$`
- HTTPS: `^https?://[^/]+/([^/]+)/(.+?)(?:\.git)?/?$`

GitLab subgroup URLs (multi-segment path) MUST return None (deferred to v1.7 per ONTOLOGY-SPEC §11).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer→URI helpers | Helpers are leaf functions; inputs come from in-process Python (RepoContext, package name strings) — no untrusted user input in Plan 02. |
| git→parse_remote_url | `parse_remote_url` receives the output of `git remote get-url origin` (Plan 05 callsite). Git remote URLs originate from `.git/config` which is developer-controlled, but worth treating as semi-untrusted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-02-01 | Denial-of-Service | parse_remote_url regex DoS via pathological input (catastrophic backtracking) | mitigate | Both regexes use bounded character classes (`[^:]+`, `[^/]+`, `[^/]+?`) and a single optional trailing group — no nested quantifiers, no overlapping alternations. Linear-time matching guaranteed. Add a parse_remote_url case for an absurdly long input (~10KB) that returns None within milliseconds if concern grows. |
| T-28-02-02 | Spoofing | Malicious remote URL crafted to inject path separators into URI (e.g., `https://x/a/b/../c`) | accept | URI helpers do not interpret URIs as filesystem paths; downstream consumers (SQL writes, file lookups) operate on independent path inputs. Risk surface is cosmetic (malformed URI strings in DB), not exploit-class. |
| T-28-02-03 | Information disclosure | `domain_uri('secret-internal-project')` leaks internal domain names if exported | accept | URIs are local to a developer's `code.db`; nothing in Phase 28 transmits URIs over a network. v1.7+ remote-sync would need to revisit. |
| T-28-02-SC | Tampering | npm/pip/cargo installs | accept | Plan installs zero packages; no Package Legitimacy Gate required. |
</threat_model>
