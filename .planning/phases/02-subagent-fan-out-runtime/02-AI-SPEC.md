## 1. System Classification

**System Type:** Multi-Agent (concurrent fan-out runtime / task dispatch infrastructure)

**Description:**
`SubagentPool` is a concurrent fan-out primitive that dispatches a batch of items to a role-bound Bedrock model in parallel, collects per-item results with partial-failure isolation, and writes a structured JSONL trace record for every invocation. It is the shared infrastructure layer that every wiki command (query, scan, lint, ingest) depends on before any command can perform parallel work. "Good" means: all N items are dispatched within the role's `max_concurrency` semaphore budget, any individual task failure surfaces in `FanOutResult.errors` without cancelling sibling tasks, and every call writes a complete trace record (role, model_id, prompt_hash, item_id, status, latency_ms, tokens_in, tokens_out) â€” even for tasks that error.

**Critical Failure Modes:**
1. **Sibling cancellation on partial failure** â€” if one task raises and `asyncio.gather` is called without `return_exceptions=True`, all in-flight sibling tasks are immediately cancelled. The entire fan-out silently loses results. This is deepagents bug #694 and the primary reason SubagentPool is built on `asyncio.gather(return_exceptions=True)` directly rather than SubAgentMiddleware.
2. **Silent trace data loss** â€” if the JSONL writer raises (disk full, bad path, permission error) inside the task wrapper and the exception is swallowed, token counts and latency data are lost permanently. Trace failures must propagate as warnings (stderr) and never silently drop records.
3. **ThrottlingException from Bedrock causing full-batch abort** â€” without a per-role `asyncio.Semaphore`, all N tasks fire simultaneously, overwhelming the Bedrock Converse API rate limit. The semaphore cap must be enforced at the pool level; individual tasks must not implement their own throttle logic.
4. **GraphRecursionError from nested subagent depth** â€” if a task itself spawns further subagents and the LangGraph `recursion_limit` is not propagated via `RunnableConfig`, execution terminates mid-batch with no partial results. The pool must pass `recursion_limit` into every task's `RunnableConfig` explicitly.
5. **`usage_metadata` field access on None** â€” `ChatBedrockConverse` returns `usage_metadata` as `None` on error responses and on some streaming paths. Accessing `.input_tokens` / `.output_tokens` without a None guard causes `AttributeError` inside the trace writer, corrupting the trace record for the current item.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling â€” AI agent infrastructure / LLM orchestration runtime

**User Population:** The `code-wiki-agent` wiki commands (`query`, `scan`, `lint`, `ingest`) are the direct callers of `SubagentPool`. Pat (the developer/operator) is the only human in the loop. There are no end-users, no customer-facing surfaces, and no multi-tenant access.

**Stakes Level:** Medium â€” data loss and silent failures corrupt wiki state permanently (missing pages, incomplete indexes). `ThrottlingException` aborts batch operations mid-run with no retry. Trace data loss blocks the Phase 4 cost-frontier eval entirely. Not life-critical, but correctness errors are irreversible without manual repair.

**Output Consequence:** `FanOutResult.successes` and `FanOutResult.errors` are acted on directly by wiki commands: successes are written to vault pages; errors trigger per-item warnings or command-level abort decisions. JSONL trace records are the sole data source for Phase 4 cost optimization â€” if records are missing or contain null token counts, the eval harness has no basis for model selection decisions.

### What Domain Experts Evaluate Against

A distributed-systems / LLM orchestration practitioner evaluating this runtime would apply the following rubric. These are not generic AI quality criteria â€” they are the specific correctness properties that matter for a concurrent fan-out dispatcher writing durable records.

---

**Dimension: Partial-failure isolation**
- Good (practitioner accepts): When 1 of N tasks raises an exception, the remaining N-1 tasks complete normally. `FanOutResult.errors` contains exactly the failing item; `FanOutResult.successes` contains exactly the N-1 successful items. No sibling is cancelled, no result is silently dropped.
- Bad (practitioner flags): Any task exception causes `asyncio.gather` to propagate immediately, cancelling in-flight siblings. The caller receives a top-level exception instead of a `FanOutResult`, losing all partial work with no per-item accounting.
- Stakes: Critical
- Source: Python asyncio docs â€” `asyncio.gather` without `return_exceptions=True` cancels the awaiting task on first exception; confirmed by deepagents issue #694 which describes exactly this production failure in SubAgentMiddleware

---

**Dimension: Concurrency throttle accuracy**
- Good (practitioner accepts): With `max_concurrency=5`, at most 5 tasks execute simultaneously against Bedrock at any moment. A burst of 50 items produces 10 sequential batches of 5. No `ThrottlingException` is raised under any tested batch size within the role's configured limit.
- Bad (practitioner flags): Semaphore is created in `__init__` (bound to a different event loop than `run_all()`), rendering the throttle ineffective or raising `RuntimeError: Task attached to a different loop`. Alternatively: tasks bypass the semaphore and all N fire simultaneously, triggering Bedrock rate limits.
- Stakes: Critical
- Source: Python asyncio docs â€” semaphore loop-attachment semantics; AWS Bedrock Converse API rate limits per model family; production pattern described in "Taming Asyncio: Production Patterns That Prevent Silent Failures" (dev.to)

---

**Dimension: Trace record completeness**
- Good (practitioner accepts): Every item dispatched â€” whether it succeeds, errors, or times out â€” produces exactly one JSONL record in the trace file. Records for failed items include `"status": "error"` and the error message. No item is traceable by id without a corresponding record.
- Bad (practitioner flags): Error-path items produce no trace record (the `_write_trace` call is inside the `try` block and is skipped on exception). Alternatively: `_write_trace` itself raises (e.g., on `OSError`) and the exception is silently swallowed, producing zero records for the current item with no diagnostic output.
- Stakes: High â€” missing records make Phase 4 cost optimization blind to a subset of invocations
- Source: LLM observability best practices, 2025 (State of Agent Engineering report): 62% of production agent deployments now require step-level traces; missing records are the #1 reason cost dashboards under-report actual spend

---

**Dimension: Token metadata accuracy**
- Good (practitioner accepts): `tokens_in` and `tokens_out` in the trace record reflect the actual `input_tokens` / `output_tokens` from `ChatBedrockConverse`'s `usage_metadata` dict. On error responses where `usage_metadata` is `None`, the fields are explicitly `null` â€” not `0`, not omitted, not an `AttributeError` crash.
- Bad (practitioner flags): `response.usage_metadata.get("input_tokens")` is called without a `None` guard. When Bedrock returns a throttling or content-filter error, `usage_metadata` is `None` and this raises `AttributeError`, corrupting the trace record and potentially crashing the trace writer mid-batch.
- Stakes: High â€” `null` token counts are recoverable; `AttributeError` crashes produce corrupt records and may abort subsequent trace writes for the remainder of the batch
- Source: `langchain-aws` source â€” `ChatBedrockConverse` sets `usage_metadata = None` on error responses; confirmed from Context7 `/langchain-ai/langchain-aws` usage_metadata field documentation

---

**Dimension: Latency measurement correctness**
- Good (practitioner accepts): `latency_ms` in the trace record reflects the wall-clock time for that specific item's `ainvoke` call, measured with `time.monotonic()` inside the semaphore, from immediately before `await task(item)` to immediately after return. A 50-item batch with `max_concurrency=5` shows 10 distinct latency groups, not one batch-total latency shared across 50 records.
- Bad (practitioner flags): Latency is measured outside the semaphore (includes queue-wait time, not just model latency). Or a single batch-start timestamp is shared across all items, making all 50 records show identical `latency_ms` values. Phase 4 haiku-vs-sonnet latency comparisons are then meaningless.
- Stakes: Medium â€” incorrect latency data misleads model routing decisions in Phase 4 but does not corrupt wiki state
- Source: Distributed systems observability practice â€” per-span timing is a baseline requirement for meaningful cost/latency analysis (Langfuse, MLflow tracing documentation, 2025)

---

**Dimension: Observability lineage**
- Good (practitioner accepts): Given a wiki command invocation, a developer can open the corresponding `.code-wiki/traces/<timestamp>.jsonl` file and identify every model call made during that invocation by `role`, `model_id`, `item_id`, and `prompt_hash`. The trace file contains no records from other invocations (one file per `run_all()` call).
- Bad (practitioner flags): Multiple concurrent `run_all()` calls append to the same trace file, interleaving records with no way to distinguish which records belong to which command invocation. Or `item_id` is always `"None"` because items lack an `id` attribute and the fallback `str(item)` produces non-unique representations.
- Stakes: Medium â€” trace lineage is necessary for Phase 4 eval reproducibility; without it, prompt hashes cannot be matched back to specific wiki commands
- Source: LLM observability best practice â€” request-scoped trace files with stable item identifiers are the minimal lineage requirement for cost attribution (Langfuse observability docs, 2025)

### Known Failure Modes in This Domain

These are production failure modes specific to concurrent task dispatch and LLM orchestration runtimes â€” not generic LLM hallucination.

1. **Semaphore loop-attachment error in test environments.** `asyncio.Semaphore` created in `__init__` (sync context) is bound to no event loop. When `run_all()` is awaited in a test that creates its own event loop (pytest-asyncio `asyncio_mode="auto"`), the semaphore's internal state is attached to a different loop, raising `RuntimeError: Task attached to a different loop`. Fix: create the semaphore inside `run_all()` on every call, binding it to the currently running loop.

2. **Thread pool exhaustion under burst concurrency.** `ChatBedrockConverse.ainvoke()` is pseudo-async: it wraps synchronous `boto3` calls in `run_in_executor`, consuming a thread per concurrent call. With `max_concurrency=10` and the default `boto3` `max_pool_connections=10`, the connection pool is exhausted before the thread pool, producing `ConnectionPool is full, discarding connection` warnings and serializing what should be parallel calls. Fix: set `max_pool_connections` on `botocore.config.Config` to at least `max_concurrency` per role.

3. **`usage_metadata` AttributeError on error responses.** When Bedrock returns a non-200 error (ThrottlingException, content filter, service unavailable), `ChatBedrockConverse` sets `response.usage_metadata = None`. Accessing `meta.get("input_tokens")` without first checking `if meta is not None` raises `AttributeError`, which â€” if not caught in `_write_trace` â€” either crashes the trace writer or propagates as an unexpected exception inside the task wrapper, masking the original Bedrock error.

4. **Silent trace data loss on write failure.** If `_write_trace` raises `OSError` (disk full, permission denied, path missing) and the exception is not caught, the exception propagates into `_run_one`, is caught by the outer `except Exception`, and the item is recorded as a `PerItemError` with the `OSError` message rather than the actual task result. The item's model response is lost. Fix: catch `OSError` inside `_write_trace`, log as `WARNING` to stderr, and return without raising â€” trace failure must never mask task success.

5. **Prompt hash collisions from truncation.** SHA-256 truncated to 16 hex chars (64 bits of entropy) has a collision probability of ~1 in 4 billion for a given corpus. For Phase 4 idempotency checks (skip re-invoking if prompt hash already traced), a collision causes a valid new prompt to be skipped as "already run." Not a Phase 2 correctness issue, but a known design tension to document for the Phase 4 implementer.

## 2. Framework Decision

**Selected Framework:** deepagents + LangChain + LangGraph (asyncio.gather pool path for SubagentPool)

**Version:** deepagents 0.6.1 / langchain 1.3.0 / langgraph 1.2.0 / langchain-aws 1.4.6

**Rationale:**
The project stack is already committed to deepagents + LangChain on AWS Bedrock. Phase 2 is infrastructure work within that stack. The specific open question â€” SubAgentMiddleware vendor subclass vs asyncio.gather pool â€” resolves to the asyncio pool path (SUB-03) because deepagents bug #694 (cancellation cascade) was merged but NOT released in 0.6.1. Building SubagentPool directly on `asyncio.gather(return_exceptions=True)` with a role-bound asyncio semaphore avoids coupling to deepagents internals in flux. Upgrade path: once deepagents ships a release with #694 included, the asyncio pool can optionally be replaced with the vendor subclass.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| SubAgentMiddleware vendor subclass | deepagents #694 (cancellation cascade) merged but not released in 0.6.1; fix touches multiple internal points â€” subclass risk until clean release ships |
| LangGraph ReAct directly (without deepagents) | Would require reimplementing built-in tools (filesystem, planning, SubAgentMiddleware) from scratch â€” deepagents IS LangGraph with batteries |

**Vendor Lock-In Accepted:** Partial â€” AWS Bedrock-only is a deliberate constraint (cost + single-provider focus); deepagents is MIT-licensed OSS with a clear LangGraph upgrade path

### Entry Point Pattern
```python
# cores/subagent-runtime/src/subagent_runtime/pool.py
import asyncio
import hashlib
import json
import logging
import time
from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Awaitable, Callable

from langchain_core.runnables import RunnableConfig

logger = logging.getLogger(__name__)


@dataclass
class PerItemError:
    item: Any
    exception: Exception


@dataclass
class FanOutResult:
    successes: list[tuple[Any, Any]] = field(default_factory=list)
    errors: list[PerItemError] = field(default_factory=list)


class SubagentPool:
    """Fan-out primitive: dispatches N items to a role-bound model in parallel.

    Usage:
        llm = make_llm("librarian")
        async def summarize(page):
            resp = await llm.ainvoke([HumanMessage(content=build_prompt(page))])
            return resp.content

        pool = SubagentPool(trace_dir=Path(".code-wiki/traces"))
        result = await pool.run_all(items=pages, task=summarize, role="librarian")
        # result.successes -> [(page, summary), ...]
        # result.errors    -> [PerItemError(item=page, exception=...), ...]
    """

    def __init__(
        self,
        trace_dir: Path,
        *,
        default_recursion_limit: int = 100,
    ) -> None:
        self._trace_dir = trace_dir
        self._default_recursion_limit = default_recursion_limit
        self._trace_dir.mkdir(parents=True, exist_ok=True)

    async def run_all(
        self,
        items: list[Any],
        task: Callable[[Any], Awaitable[Any]],
        role: str,
        *,
        model_id: str,
        max_concurrency: int,
        recursion_limit: int | None = None,
    ) -> FanOutResult:
        rlimit = recursion_limit if recursion_limit is not None else self._default_recursion_limit
        semaphore = asyncio.Semaphore(max_concurrency)
        trace_file = self._trace_dir / f"{int(time.time())}.jsonl"

        async def _run_one(item: Any) -> tuple[Any, Any] | PerItemError:
            async with semaphore:
                t0 = time.monotonic()
                try:
                    # RunnableConfig.recursion_limit key confirmed from LangGraph docs
                    _config = RunnableConfig(recursion_limit=rlimit)
                    result = await task(item)
                    latency_ms = int((time.monotonic() - t0) * 1000)
                    self._write_trace(
                        trace_file, role, model_id, item, "success",
                        latency_ms, result,
                    )
                    return (item, result)
                except Exception as exc:
                    latency_ms = int((time.monotonic() - t0) * 1000)
                    self._write_trace(
                        trace_file, role, model_id, item, "error",
                        latency_ms, None, error=str(exc),
                    )
                    return PerItemError(item=item, exception=exc)

        # return_exceptions=True: one task failure does NOT cancel siblings (deepagents #694)
        raw = await asyncio.gather(*(_run_one(i) for i in items), return_exceptions=True)

        result = FanOutResult()
        for r in raw:
            if isinstance(r, PerItemError):
                result.errors.append(r)
            elif isinstance(r, BaseException):
                # asyncio.gather itself raised (should not happen with return_exceptions=True)
                logger.error("Unexpected gather exception: %s", r)
            else:
                result.successes.append(r)
        return result

    def _write_trace(
        self,
        path: Path,
        role: str,
        model_id: str,
        item: Any,
        status: str,
        latency_ms: int,
        response: Any,
        *,
        error: str | None = None,
    ) -> None:
        # Extract token counts from ChatBedrockConverse usage_metadata
        # Fields confirmed: response.usage_metadata["input_tokens"] / ["output_tokens"]
        tokens_in: int | None = None
        tokens_out: int | None = None
        if response is not None and hasattr(response, "usage_metadata"):
            meta = response.usage_metadata  # may be None on error responses
            if meta is not None:
                tokens_in = meta.get("input_tokens")
                tokens_out = meta.get("output_tokens")

        record = {
            "role": role,
            "model_id": model_id,
            "prompt_hash": None,   # caller may set; None until computed upstream
            "item_id": getattr(item, "id", None) or str(item),
            "status": status,
            "latency_ms": latency_ms,
            "tokens_in": tokens_in,
            "tokens_out": tokens_out,
            "cost_usd": None,      # Phase 4 adds cost accounting
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        }
        if error:
            record["error"] = error
        try:
            with path.open("a") as f:
                f.write(json.dumps(record) + "\n")
        except OSError as exc:
            logger.warning("Trace write failed (data loss): %s", exc)
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `asyncio.gather(return_exceptions=True)` | Runs N coroutines concurrently; collects results AND exceptions without cancelling siblings | The core of SubagentPool â€” required for partial-failure isolation (deepagents #694 fix applied here) |
| `asyncio.Semaphore(max_concurrency)` | Limits how many tasks run simultaneously | Enforced per-role from `models.toml`; prevents Bedrock ThrottlingException under burst load |
| `RunnableConfig(recursion_limit=N)` | LangGraph config dict that caps subgraph depth | Passed to every task invocation; prevents GraphRecursionError in nested subagent trees |
| `usage_metadata` | Dict on `AIMessage` response: `{"input_tokens": N, "output_tokens": N, "total_tokens": N}` | Read in trace writer to populate `tokens_in`/`tokens_out` â€” guard against `None` before access |
| `FanOutResult` | Dataclass with `successes: list[tuple[item, result]]` and `errors: list[PerItemError]` | Returned by `pool.run_all()`; caller decides fail-fast vs degrade-gracefully based on `errors` |

### Common Pitfalls
1. **`asyncio.gather` without `return_exceptions=True`** â€” Omitting this flag means the first task exception immediately cancels all sibling coroutines. Results for completed siblings are lost. Always pass `return_exceptions=True` in `SubagentPool.run_all()`. This is exactly deepagents bug #694 â€” we fix it ourselves rather than depending on the unreleased internal patch.
2. **`usage_metadata` is `None` on error responses** â€” ChatBedrockConverse sets `response.usage_metadata = None` when Bedrock returns an error (e.g., ThrottlingException, content filter). Accessing `.get("input_tokens")` on `None` raises AttributeError. Always guard: `if meta is not None: tokens_in = meta.get("input_tokens")`.
3. **`ChatBedrockConverse.ainvoke()` is pseudo-async (wraps sync boto3 in a thread)** â€” `ainvoke()` uses `run_in_executor` under the hood because boto3 has no native async support. `asyncio.gather` still achieves concurrency (threads run in parallel), but the event loop is NOT purely non-blocking. Consequence: set `max_pool_connections` on the boto3 Config to at least `max_concurrency` per role to avoid connection pool starvation.
4. **`recursion_limit` lives in the top-level config dict, not under `"configurable"`** â€” The correct call is `RunnableConfig(recursion_limit=N)` or `{"recursion_limit": N}`. Placing it under `{"configurable": {"recursion_limit": N}}` has no effect â€” LangGraph reads from the top-level key. Confirmed from LangGraph docs: `graph.invoke(inputs, config={"recursion_limit": 5})`.
5. **Creating the semaphore outside the async context** â€” `asyncio.Semaphore` must be created in the same event loop that runs the tasks. Creating it at class construction time (`__init__`) in a sync context and then awaiting it in a different loop (e.g., in tests that create a fresh event loop) raises `RuntimeError: Task attached to a different loop`. Create the semaphore inside `run_all()` to bind it to the running loop.

## 4. Implementation Guidance

**Model Configuration:**

Seven logical roles extend the Phase 1 `models.toml`. Each role maps to a cross-region inference profile ARN; haiku handles volume/speed tasks, sonnet handles quality-critical tasks. `max_tokens` is enforced by callers via `ChatBedrockConverse(max_tokens=...)`. `max_concurrency` drives the per-role `asyncio.Semaphore` in SubagentPool.

**Core Pattern:**

The SubagentPool must be constructed with a `trace_dir` and called with per-role config loaded from `models.toml`. The caller constructs the model (`make_llm(role)`) and passes a task closure to `pool.run_all()`.

```python
# cores/subagent-runtime/src/subagent_runtime/pool.py
# (Full implementation in Section 3 Entry Point Pattern above)

# Canonical caller pattern â€” wiki command (e.g., Phase 3 query command)
from model_adapter.loader import make_llm, load_role_config
from subagent_runtime.pool import SubagentPool, FanOutResult
from langchain_core.messages import HumanMessage, SystemMessage
from pathlib import Path

async def run_librarian_fan_out(
    pages: list,
    vault_root: Path,
) -> FanOutResult:
    role = "librarian"
    role_cfg = load_role_config(role)          # reads models.toml for max_tokens, max_concurrency
    llm = make_llm(role)                       # task OWNS the model â€” pool never sees it

    async def summarize(page) -> str:
        prompt = build_librarian_prompt(page)  # caller constructs the prompt
        prompt_hash = compute_prompt_hash(prompt)
        resp = await llm.ainvoke([
            SystemMessage(content="You are a wiki librarian. Summarize concisely."),
            HumanMessage(content=prompt),
        ])
        # attach prompt_hash to response for trace writer
        resp.__dict__["_prompt_hash"] = prompt_hash
        return resp                            # return full AIMessage so pool reads usage_metadata

    pool = SubagentPool(
        trace_dir=vault_root / ".code-wiki" / "traces",
        default_recursion_limit=100,
    )
    return await pool.run_all(
        items=pages,
        task=summarize,
        role=role,
        model_id=role_cfg["model_id"],
        max_concurrency=role_cfg["max_concurrency"],
    )


def compute_prompt_hash(prompt: str) -> str:
    """SHA-256 of rendered prompt text â€” for trace deduplication."""
    import hashlib
    return hashlib.sha256(prompt.encode()).hexdigest()[:16]
```

**Tool Use:**

SubagentPool is pure infrastructure â€” no LangChain tools are bound to the pool itself. The pool's only external I/O is JSONL trace writes to disk. Individual task functions (e.g., `summarize`) may themselves call tools, but that is the caller's concern, not the pool's.

The one dependency the pool has is on `model_adapter.loader.make_llm()` (Phase 1) for model construction â€” but callers invoke `make_llm()` before constructing the task closure; the pool never receives the model object.

**State Management:**

SubagentPool is stateless between calls. State during a `run_all()` call lives in:

- `asyncio.Semaphore(max_concurrency)` â€” created inside `run_all()` to bind to the running event loop; discarded when `run_all()` returns.
- `FanOutResult` â€” in-memory; returned to the caller; not persisted by the pool.
- JSONL trace file â€” the only durable state. File is opened in append mode; each record is a single JSON line. File path: `<trace_dir>/<unix_timestamp>.jsonl`. One file per `run_all()` call.

No LangGraph checkpointer is used. No Redis or Postgres. SubagentPool is a thin concurrency layer above raw asyncio â€” persistence is the caller's responsibility.

**Context Window Strategy:**

Each role has a `max_tokens` ceiling in `models.toml`. This is the per-call output budget enforced by passing `max_tokens` to `ChatBedrockConverse` at construction time via `make_llm()`.

Input budget strategy (Phase 2 scope):
- Callers are responsible for keeping prompts within the model's context window. The pool does not truncate, chunk, or summarize input.
- For haiku roles (librarian, scanner, linter, ingestor): `max_tokens=1024â€“2048` output. Input window: 200K tokens (Claude Haiku). In practice, per-page prompts are 500â€“2000 tokens â€” no truncation needed.
- For sonnet roles (synthesizer, judge_a, judge_b): `max_tokens=2048â€“4096` output. Input window: 200K tokens (Claude Sonnet 4). Multi-page synthesis prompts may approach 10Kâ€“50K tokens â€” callers must check before dispatch and split if needed.
- Phase 3+ will add pre-flight token estimation using Bedrock CountTokens API before calling `pool.run_all()`.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs â€” independent of framework choice.

### 4b.1 Structured Outputs with Pydantic

SubagentPool itself uses plain Python dataclasses (`FanOutResult`, `PerItemError`) â€” not Pydantic â€” because the pool collects raw task results and does not validate LLM output. Pydantic structured output belongs in the **task function** that wraps the LLM call, not in the pool.

For tasks that require structured extraction (e.g., a linter task returning a `LintResult`), use `ChatBedrockConverse.with_structured_output()`:

```python
from pydantic import BaseModel, Field
from langchain_aws import ChatBedrockConverse
from model_adapter.loader import make_llm

# Define the output schema for the linter role
class LintIssue(BaseModel):
    rule: str = Field(description="Lint rule identifier, e.g. 'missing-summary'")
    message: str = Field(description="Human-readable description of the issue")
    line: int | None = Field(default=None, description="Line number, if applicable")

class LintResult(BaseModel):
    page_id: str = Field(description="Vault page identifier")
    issues: list[LintIssue] = Field(default_factory=list)
    passed: bool = Field(description="True if zero issues found")

# Bind structured output to the model â€” use method="json_schema" for Bedrock models
llm = make_llm("linter")
structured_llm = llm.with_structured_output(LintResult, method="json_schema")

async def lint_page(page) -> LintResult:
    # No retry needed when using with_structured_output â€” LangChain handles
    # Bedrock's native JSON schema enforcement. If the model returns invalid
    # JSON, LangChain raises OutputParserException â€” let it propagate to
    # pool.run_all() as a PerItemError (partial failure, not full-batch abort).
    return await structured_llm.ainvoke([
        SystemMessage(content="You are a wiki linter. Check the page and return structured issues."),
        HumanMessage(content=build_lint_prompt(page)),
    ])

# Manual retry pattern (use only if with_structured_output is insufficient):
from langchain_core.exceptions import OutputParserException
import logging

MAX_RETRIES = 2
logger = logging.getLogger(__name__)

async def lint_page_with_retry(page) -> LintResult:
    for attempt in range(MAX_RETRIES + 1):
        try:
            return await structured_llm.ainvoke([
                SystemMessage(content="You are a wiki linter."),
                HumanMessage(content=build_lint_prompt(page)),
            ])
        except OutputParserException as exc:
            if attempt == MAX_RETRIES:
                logger.error("Lint structured output failed after %d retries: %s", MAX_RETRIES, exc)
                raise  # becomes a PerItemError in FanOutResult
            logger.warning("Lint parse retry %d/%d: %s", attempt + 1, MAX_RETRIES, exc)
```

Key rules:
- Use `method="json_schema"` with `ChatBedrockConverse` â€” Bedrock's Converse API supports native JSON schema; `method="function_calling"` may not work for all Claude models on Bedrock.
- Retry at most 2 times. Surface on third failure. Log the raw LLM output at `WARNING` level on retry, `ERROR` on final failure.
- Do NOT retry inside the pool â€” retry inside the task function. The pool treats any raised exception as a `PerItemError`.

### 4b.2 Async-First Design

`ChatBedrockConverse.ainvoke()` is **pseudo-async**: it wraps the synchronous `boto3` client using `asyncio.get_event_loop().run_in_executor()`. Concurrency is real (threads run in parallel via a thread pool), but the event loop is not purely non-blocking â€” each `ainvoke` consumes a thread.

**Consequence for SubagentPool:** `asyncio.gather` achieves true parallelism here because each `ainvoke` runs in a separate OS thread. However, you must configure `boto3` with `max_pool_connections >= max_concurrency` to avoid connection pool exhaustion:

```python
import boto3
from botocore.config import Config
from langchain_aws import ChatBedrockConverse

# For roles with max_concurrency=10, set max_pool_connections=10 (minimum)
boto_config = Config(
    max_pool_connections=10,
    retries={"max_attempts": 3, "mode": "adaptive"},
    read_timeout=120,
)
# Pass via client= parameter; make_llm() should accept an optional boto_config
llm = ChatBedrockConverse(
    model="us.anthropic.claude-haiku-4-5-20251001-v1:0",
    region_name="us-east-1",
    client=boto3.client("bedrock-runtime", region_name="us-east-1", config=boto_config),
)
```

**The one common mistake:** calling `asyncio.run()` inside an already-running event loop. In pytest with `asyncio_mode = "auto"`, the event loop is already running. Never call `asyncio.run(pool.run_all(...))` in a test â€” use `await pool.run_all(...)` directly. Calling `asyncio.run()` inside a running loop raises `RuntimeError: This event loop is already running`.

**Stream vs. await for SubagentPool tasks:**

- **Use `await llm.ainvoke()`** (not `astream`) inside task functions. The pool collects discrete results per item; streaming partial tokens mid-task provides no UX benefit in a batch fan-out and prevents reading `usage_metadata` from the final `AIMessage`.
- **Exception:** If a task needs to return content as it arrives (e.g., a future streaming query command), the task is not a candidate for SubagentPool's batch pattern â€” it becomes a separate streaming coroutine at the command layer.

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation** â€” Always separate system and user prompts using `SystemMessage` and `HumanMessage`. Never concatenate role instructions into the user prompt:

```python
# CORRECT
messages = [
    SystemMessage(content="You are a wiki librarian. Summarize the page concisely in 3 sentences."),
    HumanMessage(content=f"Page: {page.title}\n\n{page.body}"),
]

# WRONG â€” mixes role instruction into user turn; degrades instruction following
messages = [HumanMessage(content=f"You are a wiki librarian. Summarize: {page.body}")]
```

**`prompt_hash` for trace deduplication** â€” compute SHA-256 of the rendered user prompt before invoking the model. Store as the first 16 hex chars to keep trace records compact. This allows idempotency checks in Phase 4 (skip re-invoking if the same prompt was already traced successfully):

```python
import hashlib

def compute_prompt_hash(user_prompt: str) -> str:
    return hashlib.sha256(user_prompt.encode()).hexdigest()[:16]
```

**`max_tokens` is mandatory** â€” Never leave `max_tokens=None` (the `ChatBedrockConverse` default) in production. Unbounded output tokens cause runaway cost and latency. Set `max_tokens` via `models.toml` per role and enforce it at model construction time in `make_llm()`. Haiku tasks: 1024. Sonnet synthesis: 4096. Eval judges: 2048.

**Few-shot examples:** For linter and judge roles that need consistent formatting, embed 1â€“2 few-shot examples directly in the system prompt (inline). Do not use dynamic retrieval for few-shot in Phase 2 â€” the prompt is deterministic and the vault corpus is not yet indexed.

### 4b.4 Context Window Management

SubagentPool dispatches items as individual LLM calls â€” there is no shared context window across items in a fan-out batch. Each task sees only its own prompt.

**Per-role token budget enforcement:**
- `max_tokens` in `models.toml` is the output ceiling. Enforce at `ChatBedrockConverse` construction: `make_llm(role)` reads `max_tokens` from `models.toml` and passes it to `ChatBedrockConverse(max_tokens=role_cfg["max_tokens"])`.
- Input budget: no automatic truncation in Phase 2. Callers must ensure prompts fit within the model's context window (200K tokens for Claude Haiku and Sonnet). For Phase 3+ fan-out over large vault pages, add a pre-flight check using the Bedrock CountTokens API before calling `pool.run_all()`.

**Detecting oversized prompts before dispatch:**
```python
# Pre-flight estimate using anthropic_bedrock token counting
# (Phase 3+ â€” not Phase 2 scope, but the pattern to follow)
import boto3

def estimate_tokens(messages: list, model_id: str, region: str = "us-east-1") -> int:
    client = boto3.client("bedrock-runtime", region_name=region)
    resp = client.count_tokens(
        modelId=model_id,
        messages=[{"role": m.type, "content": m.content} for m in messages],
    )
    return resp["inputTokenCount"]
```

**No compaction needed for SubagentPool** â€” Unlike conversational agents that accumulate message history, SubagentPool tasks are stateless per-item. There is no growing context window to compact. The entire Phase 2 context management strategy is: enforce `max_tokens` per role and let callers manage input size.

### 4b.5 Cost and Latency Budget

**Phase 2 cost tracking:** `cost_usd` is `null` in the trace schema. Phase 4 (eval harness) adds cost accounting when it has full model pricing context. Do not hardcode pricing rates in Phase 2.

**Latency capture pattern** â€” capture wall-clock latency per item using `time.monotonic()` around the `ainvoke` call (already shown in pool.py):
```python
t0 = time.monotonic()
result = await task(item)
latency_ms = int((time.monotonic() - t0) * 1000)
```

**Expected latency benchmarks** (approximate, based on Bedrock Converse API p50 in us-east-1 as of May 2026):
- Claude Haiku: 300â€“800 ms per call (1024-token output)
- Claude Sonnet: 1500â€“4000 ms per call (2048-token output)
- With `max_concurrency=10` (haiku), a 50-page fan-out completes in ~5 batches Ă— ~500ms = ~2.5s
- With `max_concurrency=5` (librarian haiku), a 50-page fan-out: ~10 batches Ă— ~500ms = ~5s

**Haiku vs. Sonnet routing decision** (implemented via `models.toml`):
| Role | Model | Rationale |
|------|-------|-----------|
| librarian, scanner, linter, ingestor | Haiku | High volume, fast, cheap; output is structured or short |
| synthesizer, judge_a, judge_b | Sonnet | Quality-critical; lower volume; slower is acceptable |

**Caching:** Phase 2 does not implement caching. Phase 4 will use `prompt_hash` from the trace record to skip re-invoking the model for identical prompts (exact-match cache on hash â†’ stored output). Semantic caching is out of scope for this project.

**boto3 connection pool cost:** Each role's Semaphore cap should match `max_pool_connections` on the boto3 Config to avoid thread contention overhead. Connection pool exhaustion (not model latency) is the dominant latency driver for burst fan-out at `max_concurrency >= 5`.

## 6. Guardrails

> This is an internal, single-developer tool with no user-facing surface. Traditional content-safety guardrails do not apply. The relevant guardrails are infrastructure correctness checks that prevent silent data loss and runaway resource use.

### Online (Real-Time)

These run inside `SubagentPool.run_all()` on every invocation. Each adds negligible overhead.

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Semaphore cap enforcement** | Every task acquisition via `asyncio.Semaphore(max_concurrency)` | Block â€” at most `max_concurrency` tasks proceed; excess tasks wait in the semaphore queue. Semaphore is created inside `run_all()` (not `__init__`) to guarantee correct event-loop binding. |
| **Recursion limit injection** | Every `task(item)` invocation site inside `_run_one` | Block â€” `RunnableConfig(recursion_limit=N)` is constructed and passed to every task call. If the task ignores it, that is the task's bug; the pool's responsibility is to always construct and pass the config. |
| **Trace writer error isolation** | `OSError` raised inside `_write_trace` | Flag â€” caught explicitly; logged as `WARNING` to stderr with the path and error message; `_write_trace` returns without raising. Task result is not masked by trace failure. |
| **`usage_metadata` None guard** | Every `_write_trace` call where `response` is not None | Flag â€” `if meta is not None` check before accessing token fields; fields default to JSON `null` on error responses rather than crashing with `AttributeError`. |

### Offline (Flywheel)

These are not automated alerts â€” they are metrics Pat reviews by inspecting trace files after real wiki command runs. Phase 4 will automate this analysis.

| Metric | How to Measure | Action on Degradation |
|--------|---------------|----------------------|
| **Trace completeness rate** | `records_written / items_dispatched` per trace file â€” parse JSONL and count lines vs. the batch size logged at `run_all()` entry | Completeness < 1.0 means `_write_trace` silently failed for some items. Investigate the WARNING log for `OSError` messages. Check disk space and trace directory permissions. |
| **Token metadata capture rate** | Fraction of success records where `tokens_in` and `tokens_out` are non-null integers | Rate < 1.0 on success records indicates `usage_metadata` is unexpectedly None on non-error responses â€” possible `langchain-aws` regression. Upgrade or pin `langchain-aws` version. |
| **Error rate per role** | `len(errors) / (len(successes) + len(errors))` per `run_all()` call, grouped by role | Sustained error rate > 10% for a role not under test suggests a Bedrock model availability issue or malformed prompt. Inspect `PerItemError.exception` messages in the trace records. |

| `items_dispatched` | Count of JSONL lines in the trace file | Confirms all items were attempted; mismatch vs. input batch size indicates silent truncation |
| `success_rate` | `count(status=="success") / total_records` per file | Pool-level health; sudden drop means Bedrock errors or task logic failures |
| `mean_latency_ms` per role | Average of `latency_ms` across records for a given role | Per-role performance baseline; use to detect model latency regression between Bedrock regions or model versions |
| `tokens_in` / `tokens_out` distribution | Min/max/mean per role across a trace file | Input prompt size hygiene; output token budget utilization; feeds Phase 4 cost modeling |
| `trace_completeness_rate` | `total_records / items_dispatched` | Must equal 1.0; any shortfall means trace writer failures occurred (check stderr for WARNING logs) |

**Alert Thresholds:** Not applicable for a personal single-developer project. There is no alerting infrastructure and no SLA. The operating principle is: investigate on any pytest failure. For production runs, inspect trace files when a wiki command produces unexpected output (missing pages, incomplete scan results, etc.).

**Sampling Strategy:** Not applicable for Phase 2. Every `run_all()` invocation is fully traced â€” 100% coverage with no sampling. All records for every item in every batch are written to the trace file. Sampling becomes relevant in Phase 4 when the eval harness runs large-scale sweeps across many models and prompts.

**Trace File Lifecycle:** Trace files accumulate in `.code-wiki/traces/`. No automatic rotation or deletion in Phase 2. Pat manually prunes old files. Phase 4 will specify a retention policy when building the eval harness that reads these files.
