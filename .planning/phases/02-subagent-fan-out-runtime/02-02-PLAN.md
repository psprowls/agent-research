## Phase Goal

**As a** wiki command author, **I want to** dispatch N items in parallel through a SubagentPool that tolerates per-item failures, enforces per-role throttle limits, propagates recursion limits, and records a complete JSONL trace of every invocation, **so that** I can build fan-out commands (query/scan/lint/ingest) on a primitive that never silently drops results or aborts a batch on a single failure.

<objective>
Deliver the SubagentPool vertical slice: end-to-end working fan-out from a caller's perspective, fully testable with mocked LLM responses. After this plan, a unit test can:

1. Construct `pool = SubagentPool(trace_dir=tmp_path / "traces")`
2. Build a `task` closure that returns `fake_llm_response` (with `usage_metadata`) for some items and raises `ValueError` for others
3. Call `await pool.run_all(items=[...], task=task, role="scanner", model_id="us.anthropic.claude-haiku-...", max_concurrency=4)`
4. Receive a `FanOutResult` with the correct split between successes and errors
5. Open the resulting `.jsonl` file and parse exactly N records, each with the 9-field schema and correct tokens_in/tokens_out (null on the error path)

Purpose: SUB-01..07, OBS-01, and the trace-write side of BED-05 all converge in this primitive. Building it with unit-test-level mocks (no Bedrock) lets us prove correctness against all five Critical Failure Modes from 02-AI-SPEC Section 1 before committing to a real-Bedrock integration test in Plan 03.

Output: `cores/subagent-runtime/src/subagent_runtime/pool.py` (full implementation), updated `__init__.py` with public exports, and `tests/test_pool.py` with the 12 unit cases from 02-VALIDATION.md.
</objective>

    item: Any

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller → `pool.run_all(items, task, role)` | `items` and `role` are caller-controlled; `task` is a caller-provided closure. Pool does not validate items. |
| `task(item)` → external (LLM or other) | Pool wraps every task call in try/except Exception; per-item failures are isolated as `PerItemError`. |
| `_write_trace` → filesystem | `trace_dir` is constructed from caller-controlled path. No symlink dereference or path traversal validation (trace_dir is single-operator-controlled). |
| `item_id` field → JSONL record | `str(item)` may contain control characters; `json.dumps` correctly escapes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | JSONL injection via `item_id` | accept | `json.dumps()` correctly escapes all control chars and quotes per 02-RESEARCH Security Domain |
| T-02-02-02 | Information Disclosure | AWS credentials embedded in error messages or item reprs | mitigate | Document in pool docstring that task closures must not include credentials in prompts or `item.id`; `_write_trace` reads only `item_id`, `error=str(exc)`, and structured fields — no secrets unless the caller leaks them |
| T-02-02-03 | Denial of Service | Trace write OSError aborts the batch | mitigate | `_write_trace` catches OSError internally; logs WARNING; never raises — covered by `test_write_trace_oserror_logged_not_raised` |
| T-02-02-04 | Denial of Service | Unbounded concurrency exhausts boto3 connection pool | mitigate | Per-role `max_concurrency` semaphore enforces ceiling; covered by `test_semaphore_caps_concurrency`. boto3 pool sizing itself is Phase 1's responsibility (make_llm in model-adapter) — Plan 03 will verify against real Bedrock. |
| T-02-02-05 | Tampering | `usage_metadata=None` causes AttributeError corrupting trace record | mitigate | Explicit None guard before accessing `meta.get(...)`; covered by `test_token_metadata_none_guard` |
| T-02-02-06 | Repudiation | Missing trace records make Phase 4 cost optimization blind | mitigate | `_write_trace` called in both try and except branches of `_run_one`; covered by `test_trace_record_completeness_success_path` and `test_trace_record_error_path` |
| T-02-02-07 | Tampering | sibling cancellation on partial failure (deepagents #694) | mitigate | `asyncio.gather(return_exceptions=True)` per 02-AI-SPEC Failure Mode #1; covered by `test_partial_failure_isolation`, `test_first_task_failure_does_not_cancel_siblings`, `test_all_tasks_fail` |
