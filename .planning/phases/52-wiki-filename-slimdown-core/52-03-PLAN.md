<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Scanner-classified `app` node → `_queries.list_apps` | `app` nodes are produced by graph-io's classification pipeline (Phase 50); the classification is gated by `_VALID_APP_KINDS`. No external user input reaches the wiki-io render path. |
| `entity-app.md` template → `_render_entity_page` | Template is a checked-in resource; loaded via `importlib.resources`. No path injection possible. |
| `describe_app` SQL query → frontmatter dict | Existing pattern; queries are parameterized via `?` placeholders in graph-io. No SQL injection surface added by Phase 52. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-03-T | Tampering | new `entity-app.md` template | mitigate | Template is a static checked-in resource under `packages/wiki-io/src/wiki_io/assets/page-templates/`; modifications require code review like any other source file. `frontmatter.load` validates YAML on read. |
| T-52-03-V | Input Validation | `scanner_frontmatter_for_node` app branch | mitigate | Reuses the existing pattern (`if d is not None`); `describe_app` returns a strongly-typed dataclass; field access is attribute-based with no string interpolation. |
| T-52-03-I | Information Disclosure | none | accept | All surfaced fields are graph-derived public metadata (language, version, etc.) already documented in `SCANNER_OWNED_KEYS`. |
| T-52-03-D | Denial of Service | adding `list_apps` to per-kind loop | accept | One additional `list_<kind>` query per `write_entities` call — bounded by the number of app nodes in the graph (typically 0-5 per repo); cost is negligible. |
| T-52-03-R | Repudiation | app entity page provenance | accept | Each rendered page has `uri:` frontmatter identifying its graph-side origin; reproducible from the scanner. |
| T-52-03-S | Spoofing | none applicable | accept | No identity layer. |
</threat_model>
