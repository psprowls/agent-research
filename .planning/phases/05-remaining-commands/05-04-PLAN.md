## Phase Goal

**As a** Pat, **I want to** run `code-wiki-agent scan` (CLI) or `wiki_scan` (MCP) against my monorepo and have the agent (a) deterministically discover packages and diff against vault state, (b) fan out scanner subagents to write meaningful stub bodies for new/changed packages in parallel via Bedrock, and (c) flag renamed/deleted packages with `stale: true` and log entries, **so that** the vault stays in sync with the codebase without manual stub authoring.

<objective>
Deliver the `scan` command end-to-end vertical slice: `commands/scan.py` (deterministic discovery + scanner fan-out + post-processing), CLI `scan` subcommand, MCP `wiki_scan` tool with progress notifications, and parity test against the existing `single-package-vault` fixture.

This is THE most complex non-lint plan in the phase. Per RESEARCH §`scan` Command (Complexity: HIGH). The two non-obvious requirements are: (1) the scanner role has `max_tokens = 500`, so the LLM generates ONLY the stub body — the `## File map` section is appended from deterministic `build_file_map()` output (Risk 5); (2) deleted packages get a `stale: true` frontmatter mutation, not deletion (D-07). Per MVP_MODE: after this plan, a real user can scan a monorepo and watch stubs appear — a capability that did not exist before.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-provided vault path → scan | File system access at vault and repo root |
| Bedrock scanner LLM response → vault page write | LLM output is written to disk as markdown |
| Existing vault page → stale-tag mutation | Mutates user files (frontmatter prepend) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | Tampering | Scanner LLM produces invalid markdown / malicious frontmatter | mitigate | `SCANNER_SYSTEM` constrains output structure; downstream lint catches invalid frontmatter on next run; stub bodies are written under known vault subdirs only |
| T-05-04-02 | DoS | Discovery walks pathological monorepo (10k packages) | accept | `max_depth` parameter caps walk; `SubagentPool.max_concurrency` bounds parallelism; user can Ctrl-C |
| T-05-04-03 | Tampering | `stale: true` prepend corrupts existing frontmatter | mitigate | Idempotent: check for existing `stale:` line before prepending; preserve other frontmatter keys |
| T-05-04-04 | Spoofing | MCP host passes attacker-controlled `vault_path` | accept | Same-user process; vault_path resolves to a real workspace anyway |
</threat_model>
