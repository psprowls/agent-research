## Phase Goal

**As a** Pat, **I want to** run `code-wiki-agent ingest source path/to/file.md` (route + summarize via ingestor subagent + file + cross-ref + log) OR `code-wiki-agent ingest work-item --frontmatter ... --body ...` (validate + file + cross-ref + log), **so that** new source files and work items land in the right vault location without manual filing.

<objective>
Deliver the `ingest` command end-to-end vertical slice: two CLI subcommands under a Typer sub-app, ONE MCP `wiki_ingest` tool with a `type` discriminator (planner's discretion per D-04, chosen for cleaner MCP tool description), `commands/ingest.py` shared layer that wraps the vault_io ports from plan-05-03, and unit tests.

The ingestor subagent (D-02) decides target page type (package/concept/adr) and writes the body — there is no deterministic routing. Cross-ref update (D-03) is scoped to `update_index(wiki)` only for v1; deeper back-ref scans are explicitly deferred per CONTEXT.md `<deferred>` ("ingest cross-ref deep linking — if too costly, scope down to index-only for v1"). This decision is documented here.

Per MVP_MODE: after this plan, a real user can ingest source files and work items end-to-end — capabilities that did not exist before.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-supplied `source_path` → extract | Reads arbitrary file the user has access to |
| User-supplied `frontmatter` YAML → _parse_frontmatter | Untrusted YAML (mitigated by yaml.safe_load in port) |
| Ingestor LLM output → vault page write | LLM produces YAML+markdown stored on disk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Tampering | Ingestor LLM produces invalid frontmatter | mitigate | `INGESTOR_SYSTEM` constrains output; downstream `lint` rejects invalid pages; `_parse_ingestor_response` falls back to slug-from-title if parse fails |
| T-05-05-02 | Tampering | LLM picks malicious `target_slug` (e.g., `../etc/passwd`) | mitigate | `slugify()` strips non-alphanumeric chars; `_route_target_path` only joins under wiki subdirs; Path.resolve() and prefix check |
| T-05-05-03 | DoS | Source file is 1GB | mitigate | `extract()` has `LARGE_FILE_BYTES` cap inherited from port |
| T-05-05-04 | Information Disclosure | Ingestor LLM has access to extracted file contents | accept | Same user context; user explicitly invokes ingest on the file; Bedrock data-plane terms apply |
</threat_model>
