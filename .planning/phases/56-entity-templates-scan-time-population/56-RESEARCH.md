## 1. The two-token rule, made concrete (D-01)

**The contradiction CONTEXT resolves:** SCAN-01 originally said "no `<...>` survives"; ENTITY-02
wants `TODO: <instructions>` to survive. D-01 splits the namespace:

- `{{var}}` (double-brace) = **scan-substituted data**. The scanner replaces these.
- `<...>` (angle) = **human-authoring instruction**. The scanner NEVER touches these.

**SCAN-01 verify is reinterpreted as: "no unsubstituted `{{...}}` token survives in a generated
entity page."** (CONTEXT D-01.) The ROADMAP SC#1 literal ("no literal `<...>` survives") is
satisfied for the *data-bearing* `<...>` because those become `{{var}}` and get substituted;
the *instruction* `<...>` inside `> TODO:` blockquotes is intentionally retained and is NOT a
violation. Downstream agents must NOT "fix" surviving `<...>` inside TODO blockquotes.

### Where `<...>` appears in the 7 entity templates today (verified by reading each file)

Every `entity-<kind>.md` body currently opens with a **data-bearing** angle heading:

| Template | Current H1 | Must become (D-01) |
|---|---|---|
| entity-package.md | `# <Package Name>` | `# {{package_name}}` |
| entity-app.md | `# <App Name>` | `# {{app_name}}` (or shared `{{name}}`) |
| entity-domain.md | `# <Domain Name>` | `# {{domain_name}}` |
| entity-plugin.md | `# <Plugin Name>` | `# {{plugin_name}}` |
| entity-dependency.md | `# <Dependency Name>` | `# {{dependency_name}}` |
| entity-repository.md | `# <Repository Name>` | `# {{repository_name}}` |
| entity-test-suite.md | `# <Test Suite Name>` | `# {{test_suite_name}}` |

**Why the H1 matters:** the frontmatter block is fully regenerated by
`merge_frontmatter`/`scanner_frontmatter_for_node` + `yaml.safe_dump` (so the `title: <...>` /
`uri: <...>` frontmatter `<...>` values never reach a generated page). But the **template body**
is used verbatim today (`_render_entity_page` does `body = template.content` with NO
substitution — entity_writer.py:489-490). So the body H1 `# <Package Name>` survives into every
generated page unless we (a) convert it to `{{package_name}}` and (b) substitute at render time.

**Instruction `<...>` to RETAIN** (examples seen in templates — these stay): `[[concepts/<concept>]]`,
`[[packages/<other-pkg>]]`, `- **<Flow name>** — \`<path/to/entry>:<line>\``, `[[adrs/<id>-<slug>]]`,
`\`<file>\``, `\`<command>\``. These live inside human-authoring guidance / TODO regions.

`entity-test-suite.md` ALREADY contains `{{PACKAGE_SLUG}}` tokens (partial prior migration) AND
`<command>`/`<file>`/`<path>` instruction angles — a live example of the two-token coexistence.

## 2. The substitution mechanism to reuse (D-02)

`init_vault.render_template()` (init_vault.py:78-94) is a plain loop:

```python
for key, value in variables.items():
    text = text.replace("{{" + key + "}}", value)
```

**Plan target (D-02):** add the same `{{...}}` substitution to
`entity_writer._render_entity_page()` (entity_writer.py:482-500), which today does:

```python
template = frontmatter.load(template_path)
body = template.content          # ← verbatim, NO substitution
yaml_block = yaml.safe_dump(frontmatter_dict, ...)
rendered = f"---\n{yaml_block}\n---\n{body}".rstrip("\n") + "\n"
```

The substitution must run on `body` (the frontmatter is built separately from the dict, so
`{{...}}` tokens only need substituting in the body). Build a `variables: dict[str, str]` from
the node + its derived frontmatter, then loop `body = body.replace("{{"+k+"}}", v)` BEFORE
composing the rendered string. Do NOT add Jinja.

**Signature consideration:** `_render_entity_page(template_path, frontmatter_dict)` currently
takes only the merged frontmatter dict. The substitution variables (name/slug/kind/version/etc.)
must be derived from the node and/or the `frontmatter_dict`. Cleanest options for the planner:
- (a) Pass the `node` and `kind` into `_render_entity_page` and build `variables` inside it, OR
- (b) Build a `variables` dict at the `write_entities` call site (where `node`, `kind`, `slug`,
  and `scanner_fm` are all in scope — entity_writer.py:730-738) and pass it as a 3rd arg.

Option (b) keeps `_render_entity_page` close to its current shape and puts variable derivation
next to `scanner_frontmatter_for_node` where node data is already read. Either is fine (D-04
leaves the exact var set to discretion); the planner should pick one and be concrete.

### D-03: unfilled `{{var}}` → TODO marker, never a raw `{{var}}` or empty string

If a variable has no value (e.g. missing version/description), substitute a
`> TODO: <add ...>` marker (D-12 blockquote format) so (a) SCAN-01 stays satisfied (no surviving
`{{...}}`) and (b) the gap is visible. Practically: after building `variables`, any `{{var}}` the
template references that has no node-derived value resolves to a TODO string instead of being
left unsubstituted. The substitution loop must cover EVERY `{{var}}` introduced by the migrated
templates (D-04) — a missed token is a SCAN-01 failure.

**Self-documenting code (Pat preference):** comment the `{{...}}`-data-vs-`<...>`-instruction
distinction at the substitution site so a future reader doesn't collapse the two syntaxes.

## 3. The `summary:` field (SCAN-02) — fill-when-empty, NOT scanner-owned (D-05/D-06/D-07)

### Source: `node.attrs["description"]`, read uniformly (D-05)

`scanner_frontmatter_for_node(conn, kind, node)` (entity_writer.py:528-588) builds the scanner
frontmatter dict per kind. Today it reads relation lists from `describe_*` and some attrs from
`node.attrs`. The `node` passed in is a `NodeRecord` (from `list_packages`/`list_apps`/
`list_domains`/etc.) whose `.attrs` carries the parsed `attrs_json` (queries._load_node_record,
queries.py:185-200, re-injects the `uri` column into attrs). So `node.attrs.get("description")`
is the **uniform** read across all kinds — D-05.

**Today's gap:** only **domains** reliably populate `attrs["description"]` (domains.py:258-259
copies `description` from domain attrs into the node attrs; `DomainDescription.description` exists
at queries.py:90-95). Packages/apps do NOT carry a description in node attrs yet → D-06.

### D-06: the cross-package graph-io change (the ONLY graph-io touch)

**Two small additions in `packages/graph-io/src/graph_io/packages.py`:**

1. **Parse it.** `_read_pyproject()` (packages.py:46-72) builds the manifest `info` dict but does
   NOT currently include `description`. Add `"description": project.get("description", "")` to
   the returned dict (line ~65-72). (Optionally mirror in `_read_package_json()` at ~93-99 with
   `data.get("description", "")` for JS packages — low cost, keeps the source uniform; planner's
   discretion.)

2. **Store it on the node.** In `refresh()`, the per-manifest `attrs` dict is built at
   packages.py:159-164:
   ```python
   attrs: dict[str, Any] = {
       "version": info["version"],
       "dependencies": info["dependencies"],
       "language": info["language"],
       "uri": new_uri,
   }
   ```
   Add `"description": info.get("description", "")` here. This region is the **NODE-EMISSION**
   block (the `nodes = [GraphNode(... attrs=attrs)]` at 193-201 and the opposite-kind in-place
   UPDATE at 176-191 both read this `attrs`). It is a DIFFERENT region from Phase 55's edits
   (see §5). Empty/absent description → SCAN-02's TODO fallback (D-03 style) at write time.

**No `describe_*` change needed:** because `scanner_frontmatter_for_node` reads
`node.attrs["description"]` directly off the `NodeRecord` (D-05), and `list_packages`/`list_apps`
return NodeRecords with the full attrs_json, the description flows through without touching
`describe_package`/`describe_app`/`PackageDescription`/`AppDescription`. (The planner may still
choose to surface it via describe_* for symmetry, but it is not required and adds surface area.)

### D-07: `summary:` ownership = fill-when-empty (a THIRD category)

`summary` is NEITHER overwrite-every-scan (`SCANNER_OWNED_KEYS`) NOR purely human-owned. It must
be **special-cased in `merge_frontmatter`** (entity_writer.py:357-390):

- `merge_frontmatter(existing, scanner_update)` today: (1) uri, (2) kind, (3) `SCANNER_OWNED_KEYS`
  alphabetical non-empty from `scanner_update`, (4) human keys from `existing` preserved verbatim.
- **DO NOT add `summary` to `SCANNER_OWNED_KEYS`** (entity_writer.py:104-137) — that set is
  full-replacement-every-scan and would clobber a human-edited summary.
- **Special-case:** if `existing` has a non-empty `summary`, keep it (it falls through the
  current step-4 human-key preservation already — but `summary` would then never get filled when
  absent). If `existing` has no `summary` (or it's empty), write the scanner-derived summary
  (from `scanner_update["summary"]`, sourced from `node.attrs["description"]`, or the D-03 TODO
  fallback when description is empty).
- **Where summary enters `scanner_update`:** add `summary` to the dict built by
  `scanner_frontmatter_for_node` (read `node.attrs.get("description")` uniformly, D-05). Because
  `summary` is NOT in `SCANNER_OWNED_KEYS`, the current merge loop (step 3) will SKIP it — hence
  the explicit special-case in `merge_frontmatter` is required to place it in the output.

**Verify (D-07):** re-scan PRESERVES a human-set `summary`; re-scan FILLS an empty/absent one
from the description. Both directions need a test.

**Self-documenting code:** comment the `summary` fill-when-empty special-case so a future reader
doesn't "simplify" it into `SCANNER_OWNED_KEYS` (which would silently clobber human edits).

### Downstream (not modified here): index_generator.py reads `summary`

`index_generator.py:~351,380,569` already reads `summary` from entity pages for IDX-03 (Phase
57). `summary:` must LAND in this phase first. Phase 56 does not modify index_generator.

  human-authored sections render as `> TODO: <...>` blockquotes (not empty headings / dead links).

  grep for legacy-dir path references returns no live code reference; generated pages have no dead
