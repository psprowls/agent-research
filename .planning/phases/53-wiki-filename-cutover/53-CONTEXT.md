<domain>
## Phase Boundary

**Phase 53 is dramatically narrower than originally roadmapped.** No `cg migrate-vault` extension. No wikilink rewriter. No cross-vault mapping derivation. No atomic-cutover-commit choreography. The justification (delivered explicitly by the user during discussion): there are no production wikis in existence; Pat is the sole developer and sole user of graph-wiki; vault regeneration is a manual local operation, not a deployable migration.

What Phase 53 IS:

1. **Source-code cleanup** — delete `encode_slug` and `decode_slug` (Phase 52 D-09 deferral) and any remaining call sites.
2. **Test-fixture closure** — confirm `packages/wiki-io/tests/fixtures/round-trip-vault/` uses the new short-filename scheme (Phase 52 may have handled some/all of this; Phase 53 closes any gap).
3. **Verification gate** — Pat manually deletes the `~/Personal/graph-wiki/agent-research` vault, runs `cg update --full` to regenerate from scratch, then `graph-wiki-agent scan` and confirms `wiki/index.md` reflects new short filenames with no scan errors.
4. **Roadmap update** — strike SC #1 and SC #2 (the `cg migrate-vault` criteria) from §53; reframe SC #3 / SC #4 as verification observations rather than new code criteria.

In scope: source deletions in `wiki-io`; fixture closure; ROADMAP.md §53 edit; verification UAT.

Out of scope: any migration command, wikilink rewriter, manifest extension, atomic-cutover commit, or `wiki/package-family/` directory removal logic. The vault is regenerated wholesale — these all become moot.

</domain>

<decisions>
## Implementation Decisions

### Phase 53 scope reshape — no migration, no rewriter, no cutover command

- **D-01: No `cg migrate-vault` extension.** The existing `cg migrate-vault` command stays as Phase 46 left it (v1.8 entity-restructure migration). Phase 53 does NOT extend it with a v1.9 migration, does NOT add a migrations-dispatch package, does NOT extend the `.graph-wiki/manifest.json` schema. Rationale (locked by user during discussion): there are no production vaults, no other users, no deployable migration scenario. Vault regeneration is local + manual.
- **D-02: No wikilink-rewriter for filename cutover.** The Phase 46 rewriter machinery is not re-invoked for Phase 53. Curated-lane `[[old-long-slug]]` references in the existing vault are not algorithmically rewritten — they get cleaned up implicitly when Pat regenerates the vault from scratch (any human-authored `[[…]]` references in `/concepts/`, `/adrs/`, etc. that pointed to old long filenames will need manual cleanup at that point; the user accepts this cost given the small surface area of curated content).
- **D-03: No atomic cutover commit on the vault side.** The vault is regenerated by `cg update --full` from scratch; the new state lands as ordinary scan output, not as a single migration commit. No `.graph-wiki/manifest.json` migration marker is written.

### Source-code cleanup

- **D-04: Delete `encode_slug` and `decode_slug` from `packages/wiki-io/src/wiki_io/entity_writer.py`.** Both functions become dead after Phase 52's write path stops calling them and after Phase 53 confirms no other consumer remains. Delete the function bodies, their docstrings, and the associated `_ADMITTED_URI_PREFIXES` frozenset if (and only if) `_URI_PREFIX_BY_KIND` is the sole remaining consumer of that set.
- **D-05: Update any remaining call sites.** Researcher to grep `packages/` for `encode_slug` / `decode_slug` references after Phase 52 ships. Likely candidates: `link_rewriter.py`, scanner-side decode operations, ingest path. Each call site either (a) gets rewritten to read `frontmatter.uri` directly per Phase 52 D-08, or (b) is itself dead and gets deleted along with the surrounding function/block.
- **D-06: Drop `_ADMITTED_URI_PREFIXES` if orphaned.** This frozenset was the validation surface for `decode_slug` (entity_writer.py:87). Once decode_slug is gone, the set is orphaned. Delete unless a non-decode consumer is found (unlikely).

### Test-fixture closure

- **D-07: Phase 53 closes any remaining fixture gaps.** Phase 52 plan 04 (optional in 52-CONTEXT next_steps) may have already updated `packages/wiki-io/tests/fixtures/round-trip-vault/` for the new short-filename scheme. Phase 53 verifies that the fixture is consistent and patches anything Phase 52 left behind. Includes the `vocab.index.json` / `vocab.tokenizer.json` regeneration if filenames flow into those serialized indices.

### Verification gate (manual UAT by user)

- **D-08: Manual vault regeneration verification.** Pat performs (off-phase, after code lands):
  1. `rm -rf ~/Personal/graph-wiki/agent-research/wiki/{packages,dependencies,domain,plugin,test-suites,app}/` (or `rm -rf` the entire vault directory if that's cleaner).
  2. `cg update --full` to regenerate the graph + entity files.
  3. `graph-wiki-agent scan` to populate `wiki/index.md`.
  4. Inspect `wiki/index.md` and confirm short-filename entries (`pkg_eval-harness`, `app_graph-wiki-agent`, `dep_langchain-aws`, etc.).
  5. Spot-check 2-3 entity files exist with expected short filenames; spot-check no `pkg__org__repo__name`-style files remain.
- **D-09: UAT is recorded in `53-UAT.md`** after Phase 53 code-cleanup plans complete. UAT findings drive the phase verification report. No automated verification beyond the standard test suite — the test suite covers `short_filename` behavior (Phase 52) and entity_writer correctness; the manual UAT is the real-world end-to-end check.

### Roadmap update

- **D-10: ROADMAP.md §Phase 53 edit.** Strike SC #1 (`migrate-vault` atomic rewrite) and SC #2 (idempotency via manifest marker). Reframe SC #3 (`generate_index` writes new short filenames) as a verification observation that flows from Phase 52's `write_entities` correctness. Reframe SC #4 (vault re-scanned without errors) as the Phase 53 manual UAT gate. WIKI-FN-05 and WIKI-FN-06 requirement text in REQUIREMENTS.md is similarly reframed (or marked as "satisfied trivially via manual regen").

### Claude's Discretion (left to planner)

- Whether to mark `WIKI-FN-05` / `WIKI-FN-06` as "Withdrawn — superseded by manual regen" in REQUIREMENTS.md vs. rewriting them to match the new shape. Default: rewrite to verification-language ("`generate_index()` writes short filenames after `cg update --full`; verified manually") so the satisfaction count stays accurate.
- Whether the ROADMAP.md and REQUIREMENTS.md edits are a single Phase 53 plan (e.g., `53-01-PLAN.md: roadmap-and-requirements-reshape`) or folded into the cleanup plan. Default: single dedicated plan for the docs reshape — keeps git history clean and reviewable.
- Whether to leave `cg migrate-vault` documentation intact (it still works for the v1.8 case it shipped for) or annotate it as "historical / v1.8 only". Default: leave intact — the command is functional and the docstring already references v1.8 specifically.
- Whether the `~/Personal/graph-wiki/agent-research` vault regen happens before or after Phase 53 source code merges to main. Default: after — the cleanup PR lands first, then Pat regenerates, then UAT is recorded.

<code_context>
## Codebase Context

### Reusable patterns
- **Function deletion + call-site rewrite** is a standard cleanup pattern in this repo (mirrors Phase 51's `_SLUG_ONLY_RE` / LIB-003 removal). The grep-driven approach is:
  1. `grep -rn "encode_slug\|decode_slug" packages/` — full list of references.
  2. For each reference, decide: rewrite to `frontmatter.uri` read (D-08 from Phase 52) or delete the surrounding block.
  3. Delete the function bodies.
  4. Final verification grep returns zero hits in `packages/` (excluding planning/).

### Integration points
- **No new integration**. Phase 53 is a removal phase. The new short-filename write path (Phase 52) is already the live path; Phase 53 just shaves off the dead bidirectional-slug machinery.
- **ROADMAP.md / REQUIREMENTS.md edits are themselves the deliverable** for D-10. No code changes in those files — they're markdown.

### Known sharp edges
- If Phase 52's plan-04 fixture updates didn't happen (per its "optional" framing), Phase 53 has to do them. Researcher confirms fixture state on entry.
- `link_rewriter.py` may have its own `decode_slug`-style logic distinct from `entity_writer.decode_slug`. Phase 52 D-08 only addressed the entity_writer path; link_rewriter may still need its own treatment. Researcher to surface.
- Phase 51 D-01's deferred `wiki/package-family/` deletion is satisfied trivially by manual vault regen (D-08). No source code change needed for that deferred item.

</code_context>
