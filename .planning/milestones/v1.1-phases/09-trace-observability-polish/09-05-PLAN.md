<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-supplied `<file>` path → renderer parse loop | The renderer reads arbitrary JSONL. Malformed JSON is already caught per-line. New surface: a record claiming `schema_version: 9999` or a record with no `schema_version` must not destabilize the renderer. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-12 | Tampering | A maliciously crafted trace file claiming `schema_version: 999999` to make the warning string very long | mitigate | The warning string uses `f"warning: trace schema_version {N} is newer than supported (1); rendering best-effort"` — the integer is formatted directly. Worst case: a multi-digit integer adds a few bytes; no buffer overflow possible in Python. Acceptance: Task 2 test `test_newer_version_warns_lenient` uses 99 and asserts substring presence; the implementation is integer formatting only — no further mitigation needed. |
| T-09-13 | Information Disclosure | The v0 warning includes the file path, which may contain sensitive directory names | accept | The user supplied the path; echoing it back is expected behavior consistent with the existing `f"trace file not found: {file}"` error path already in `cli.py`. No new exposure. |
| T-09-14 | Denial of Service | A trace file with 10⁶ unversioned records | accept | The one-shot per-file flag ensures the warning is emitted once regardless of record count; parsing cost remains O(N) (same as today). No regression. |
| T-09-15 | Tampering | `schema_version` field with non-integer value (e.g., string "1.0" or null) | mitigate | Implementation guard `isinstance(record["schema_version"], int)` in the newer-version branch ensures a non-integer doesn't propagate into the comparison. A string `schema_version` is silently ignored (rendered best-effort) — consistent with the lenient-consumer policy. |
</threat_model>
