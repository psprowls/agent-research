<objective>
Close UAT gap G5 — the scanner divergence test was skipped because `round-trip-vault` has no resolvable monorepo sibling, so `run_scan` returned `result.added + result.updated == []` and `_produce_scanner_outputs` short-circuited via `pytest.skip`. The minor-severity gap was logged as a follow-on.

Per the UAT note: the issue is fixture-design, not a Phase 6 runtime defect. The fix adopts the simpler of the two options the planner was offered: **adapt the scanner test to operate on a known-good repo fixture** (the `cores/eval-harness` package itself, which is already what the helper docstring claims to use) by threading a `repo_path` override through `run_scan` to `discover_workspaces`, bypassing the cwd-based fallback.

This is preferred over extending `round-trip-vault` with a monorepo sibling because (a) `cores/eval-harness` is already a real, stable, in-repo uv workspace member with a valid `pyproject.toml`, `src/`, and tests — exactly the shape `discover_workspaces` is designed for; (b) it does not bloat the wiki-io fixture which is shared with other tests.

Output: One signature extension to `run_scan` (no behavior change when callers don't pass `repo_path`), one helper update, one unit test, one live-Bedrock baseline re-record. The baseline re-record requires GRAPH_WIKI_RUN_EVAL=1 and runs against live Bedrock — Pat runs it; the executor cannot.
</objective>
