## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM tool-call response → ProposedDomain | LLM output crosses into our type system; even with strict tool schema, the LLM could return type-violating values (e.g., string for `confidence`). |
| LLM-proposed packages → graph state | Untrusted package names from LLM could reference nonexistent packages or attempt path traversal in downstream YAML consumption. |
| `proposed_edges` → cycle detection algorithm | Adversarial LLM could craft edges to trigger pathological cycle-detection behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-48-01 | Tampering | `_parse_tool_call` and downstream YAML emission | mitigate | (a) D-05 Bedrock tool-use schema enforces type at the LLM boundary. (b) `_strip_unknown_packages` (D-09) drops any LLM-proposed package not in `list_packages` — defense in depth against hallucination and path-traversal-style strings. (c) `_strip_cycle_edges` is the only consumer of LLM-proposed `parent` field and uses iterative DFS with bounded iteration (≤ N edges per pass, ≤ N passes) — no recursion or unbounded loops. (d) `yaml.safe_dump` (D-15) prevents YAML-injection on the output side. |
| T-48-SC | Tampering | npm/pip/cargo installs | accept | No new third-party dependencies (D-25). Existing dependencies (`yaml`, `langchain-aws`, `pytest`) are already audited in earlier phases. Supply-chain checkpoint not applicable here. |
