## Q4 Answer: Where Does Relation-Frontmatter Generation Slot In?

This is the load-bearing question. The concrete answer follows from reading `scan.py` and `scan_monorepo.py` in full.

### The existing scanner stages (named by their role in `run_scan`)

| Step | Location | What it does |
|------|----------|--------------|
| Step 1 | `scan.py:run_scan` | Resolve wiki + repo paths |
| Step 1.5 | `scan.py:run_scan` | Pre-scan `cg update` (graph rebuild) |
| Step 1.6 | `scan.py:run_scan` | Open read-only graph conn |
| Step 2 | `scan.py:run_scan` | Read layout block from `wiki/CLAUDE.md` |
| Step 3 | `scan_monorepo.py:discover_workspaces` | Walk repo; emit workspace dicts |
| Step 3.5 | `scan.py:run_scan` | Decorate workspace dicts with graph URIs and domain from conn |
| Step 4 | `scan_monorepo.py:build_file_map` | Build file maps per workspace |
| Step 5 | `scan_monorepo.py:_load_existing_pages` | Load existing vault pages (discovers `apps/`, `packages/`, `domains/*/packages/`) |
| Step 6 | `scan_monorepo.py:attach_changed_files` | Git diff since last sync commit |
| Step 7 | `scan_monorepo.py:compute_diff` | Compute new / renamed / deleted / unchanged |
| Step 8 | `scan_monorepo.py:compute_state_gate` | Git state gate |
| **Step 9** | **`scan.py:SubagentPool.run_all`** | **LLM fan-out: each package → `generate_stub` → `build_stub_prompt` + scanner LLM → raw markdown** |
| **Step 10** | **`scan.py:run_scan`** | **Write stub pages to disk (`wiki_relative_path`)** |
| Step 11 | `scan.py:run_scan` | Stale-tag deleted/renamed vault pages |
| Step 12 | `scan_monorepo.py:regenerate_dependencies_index` + `update_index.py:update_index` | Regenerate `dependencies/index.md` and `wiki/index.md` |
| Step 13 | `scan.py:run_scan` | Final log entry |

**Step 9+10 is the wiki-page-producing stage.** The `generate_stub` coroutine takes a workspace dict, calls the scanner LLM, and writes the result to `pkg.get("wiki_relative_path")` (e.g. `packages/graph-io/overview.md` or `domains/billing/packages/auth/overview.md`).

### The routing that determines where a page lands

`_wiki_relative_path_for` in `scan_monorepo.py` determines the path:

- `type == "app"` → `apps/<name>/overview.md`
- `domain` key set → `domains/<domain>/packages/<name>/overview.md`
- otherwise → `<vault_dir>/<name>/overview.md` (defaults to `packages/`)

Step 3.5 is what injects the graph-derived `domain` into the workspace dict before routing.

### Q4 answer: REPLACE, not augment

Relation-frontmatter generation does not run after Step 10. It **replaces** Steps 9+10 for entities that have a graph URI. The new entity-page writer (`wiki_io/entity_writer.py`) is a deterministic function: `graph_node → frontmatter dict + skeleton body → entity page at wiki/entities/<slug>.md`. No LLM is involved in the page-write path for graph-derived entities.

The LLM scanner role shifts: instead of producing the full page body, it produces the **prose narrative** section of the entity page (the "Purpose" / "Scope" paragraphs). The relation frontmatter (edges as structured YAML) is written deterministically from the graph, not generated by the LLM.

In the new pipeline, Step 9+10 becomes:

1. **Entity page write (deterministic):** for each admitted graph node (package, domain, test-suite, repository, etc.) — build frontmatter from graph queries + write skeleton if page is new. Merge frontmatter-only update if page exists (whitelist of scanner-owned keys).
2. **LLM narrative enrichment (fan-out):** for new pages or pages whose graph-derived structural data changed — invoke scanner LLM to write/update the prose sections. Existing prose sections that were human-authored survive.

## New Files Map

### `packages/wiki-io/src/wiki_io/entity_writer.py`

**Purpose:** Deterministic entity-page writer. The only place that writes `wiki/entities/*.md`.

**Responsibilities:**
- Accept a graph conn + list of admitted node URIs
- For each node, call the appropriate `graph_io.queries.*` function to build relation frontmatter
- Determine the entity page slug from the URI (see slug derivation below)
- On create: write full skeleton with populated frontmatter + prose placeholder sections
- On update: merge scanner-owned frontmatter keys only; preserve all other keys and all prose body
- Return a report of `(created, updated, deleted)` counts

**Whitelist of scanner-owned frontmatter keys (per kind):**

```
All kinds: uri, kind, graph_name, last_scan_at
package:   domains, depends_on, test_suites, entry_points, language, version
domain:    parent_domain, sub_domains, packages, test_suites, depends_on
test-suite: tested_packages, tested_domains, file_count, suite_kind
repository: package_count, packages
dependency: used_by, versions_in_use, ecosystem  (sourced from workspace discovery, not graph)
plugin:    language, ecosystem
```

Human-authored keys (`summary`, `status`, `last_reviewed`, `tags`, custom prose) are never touched.

**Slug derivation from URI:**

| URI | Slug | Entity page path |
|-----|------|-----------------|
| `pkg:agent-research/graph-io` | `graph-io` | `entities/graph-io.md` |
| `domain:billing` | `domain-billing` | `entities/domain-billing.md` |
| `suite:agent-research/graph-io/unit` | `suite-graph-io-unit` | `entities/suite-graph-io-unit.md` |
| `repo:agent-research` | `repo-agent-research` | `entities/repo-agent-research.md` |

Collision policy: if two URIs produce the same slug, append `-2` suffix and log a warning. This should be rare given the URI scheme's structure.

### `packages/wiki-io/src/wiki_io/index_generator.py`

**Purpose:** Generate `wiki/index.md` with domain-first + by-kind structure. Fully deterministic from graph + entity pages.

**Responsibilities:**
- Accept a graph conn + wiki path
- Build domain sections: for each domain (ordered by `domain_contains_domain` tree), list contained packages/test-suites/dependencies as wikilinks
- Build global by-kind sections: all packages, all plugins, all dependencies, all test-suites, all repositories
- Entities appear in both views
- Cross-cutting packages (zero `belongs_to_domain` edges) listed in a final "Cross-cutting" section
- **Q5 answer: the index is fully scanner-generated.** Not a human-authored shell with injection blocks — it is regenerated in full each scan. Human notes belong on entity pages, not on the index.

### `packages/wiki-io/src/wiki_io/link_rewriter.py`

**Purpose:** One-shot inbound-link migration (Q3 resolution).

**Responsibilities:**
- Accept `wiki` path + a mapping `{old_wikilink: new_wikilink}`
- Walk `wiki/concepts/`, `wiki/adrs/`, `wiki/architecture/`, `wiki/work/`, `wiki/sources/`
- Rewrite `[[packages/<name>/overview]]` → `[[entities/<slug>]]` etc.
- Write the mapping to `wiki/.migration-log.json` for audit
- Idempotent: already-rewritten links are not double-processed
- **Runs once at cutover, not on every scan**

### `packages/graph-io/src/graph_io/cli/q_domain_clusters.py`

**Purpose:** `cg domain-clusters` command — import-graph clustering without LLM.

**Responsibilities:**
- Accept `--repo`, `--workspace`, `--min-cluster-size N` (default 2), `--fmt human|json`
- Query: for each pair of packages, count shared import edges via the import graph
- Apply a simple community-detection heuristic (e.g., connected components above import-count threshold)
- Emit candidate domain clusters as `{cluster_id, packages: [...], reason: "import_graph"}`
- Independently testable with no LLM dependency

**Integration with `graph_io/cli/main.py`:** register as `domain-clusters` subcommand alongside existing 25 subcommands.

### `agents/graph-wiki-agent/src/graph_wiki_agent/commands/propose_domains.py`

**Responsibilities:**

## Relation Frontmatter: Coexistence With Human-Authored Keys

The scanner-owned key whitelist is enforced by `entity_writer.merge_frontmatter`:

```python
def merge_frontmatter(existing: dict, scanner_update: dict, scanner_owned: set[str]) -> dict:
    # Human-authored keys survive untouched
    merged = dict(existing)
    # Scanner-owned keys are always overwritten
    for key in scanner_owned:
        if key in scanner_update:
            merged[key] = scanner_update[key]
    return merged
```

Any key NOT in `scanner_owned` is never touched by the scanner, even if the scanner computed a value for it. This means:
- `summary`, `status`, `last_reviewed`, `tags` → always human territory
- `uri`, `kind`, `domains`, `depends_on`, `test_suites` → always scanner territory
- Conflict resolution: scanner wins only for scanner-owned keys, human wins for everything else

## Coexistence and Cutover Story

The transition is **a wipe-and-rebuild**, not an in-place migration. The existing `~/Personal/graph-wiki/agent-research` vault is disposable per user direction (2026-05-26).

**Hard switchover at one commit.** There is no period where both old page-type directories and `entities/` coexist serving the same content. The cutover commit:

1. Runs `entity_writer.write_entities` to populate `wiki/entities/`
2. Runs `link_rewriter.rewrite_links` to update inbound wikilinks in curated lanes
3. Removes `wiki/packages/`, `wiki/apps/`, `wiki/domains/*/packages/` directories
4. Removes `wiki/dependencies/index.md` (folded into entity pages + new index)
5. Regenerates `wiki/index.md` via `index_generator.generate_index`

**During development** (before cutover): old and new code coexist. The entity writer is a new function in `wiki_io/entity_writer.py` that can be unit-tested independently against a test graph. The scanner is not modified until the cutover phase.

**Curated lanes are fully preserved.** `wiki/concepts/`, `wiki/adrs/`, `wiki/architecture/`, `wiki/work/`, `wiki/sources/` are untouched by this milestone. Their inbound wikilinks to old package paths get rewritten by `link_rewriter.py` at cutover, but their content is not touched.

## Suggested Build Order

The dependency graph between target features determines ordering:

### Foundation block (must go first)

**Feature 3 (URI-keyed entity pages) before everything else.** The slug-derivation logic and frontmatter schema must be settled before templates or writers are built. This is a pure design decision, not code.

**Feature 2 (per-kind templates) in parallel with Feature 3.** Templates and entity writer are siblings; neither depends on the other being done first.

**Feature 4 (relation frontmatter / entity writer) next.** Requires the URI scheme (Feature 3) and templates (Feature 2) to be settled. This is `wiki_io/entity_writer.py` — the core of the restructure.

### Index block (depends on entity writer)

**Feature 5 (domain-first + by-kind index) after Feature 4.** `index_generator.py` queries the same graph queries that `entity_writer.py` uses; it can reuse them. Requires entity pages to exist to generate wikilinks.

### Scanner wiring block (depends on entity writer)

**Feature 1 (entities/ lane):** Completing the entities/ lane requires wiring `entity_writer` into `scan.py` (Steps 9a/9b). This is the point where the existing scanner is modified. Do this after `entity_writer.py` is tested in isolation.

**Feature 6 (hard-delete reconciliation):** Implemented inside `entity_writer.write_entities` as part of Feature 4 logic. Not a separate phase.

### Cutover block (depends on all above being complete)

**Feature 7 (one-shot migration):** `link_rewriter.py`. This runs once, so it can be written and tested last before the cutover commit. Depends on knowing the final entity slug format (Feature 3).

**Feature 8 (curated lanes preserved):** Not a new code feature — it is the constraint that the link rewriter (Feature 7) preserves and the entity writer does not touch curated lanes. Validated by the rewriter test.

### Inference block (independent of wiki restructure)

**Feature 9 (import-graph clustering / `cg domain-clusters`):** Fully independent of Features 1–8. Lives in `graph_io/cli/q_domain_clusters.py`. Can be built in parallel or after any order.

**Feature 10 (LLM-proposed domain groupings / `propose-domains`):** Depends on Feature 9 (`cg domain-clusters`) being available. Does NOT depend on the wiki restructure.

### Recommended phase sequence

```
Phase A: URI scheme + slug derivation (design) + per-kind templates
Phase B: entity_writer.py (deterministic write + merge + hard-delete)
Phase C: index_generator.py (domain-first + by-kind)
Phase D: scan.py integration (wire entity writer into run_scan Steps 9a/9b/11/12)
Phase E: link_rewriter.py + cutover commit
Phase F: cg domain-clusters (q_domain_clusters.py) — can overlap with B-E
Phase G: propose-domains command — depends on Phase F
```

Phases B and F can run in parallel. Phases C and D cannot start until B is complete and tested.

## Anti-Patterns to Avoid

### Anti-Pattern 1: LLM writes frontmatter

**What it would look like:** Asking the scanner LLM to generate the `uri:`, `domains:`, `test_suites:` fields.

**Why it's wrong:** LLMs hallucinate field values, produce inconsistent YAML, and the graph is the single source of truth for these fields. Every time the LLM regenerates them, there will be drift.

**Do this instead:** Entity writer populates all scanner-owned frontmatter deterministically from `graph_io.queries.*`. LLM writes prose only.

### Anti-Pattern 2: Rebuilding the whole index by scanning page frontmatter

**What it would look like:** `index_generator.py` walking all `.md` files in `entities/` and parsing frontmatter to reconstruct domain membership.

**Why it's wrong:** The graph already has this information. Parsing entity page frontmatter is a second source of truth for domain structure that can drift from the graph.

**Do this instead:** `index_generator.py` queries `graph_io.queries.list_domains`, `list_packages`, `tests_for_package` etc. directly. Entity page files are used only for the wikilink targets, not for domain membership data.

### Anti-Pattern 3: Stale-tagging entity pages instead of hard-deleting

**What it would look like:** Keeping entity pages for deleted packages with `stale: true` frontmatter (the current pattern for old-style package pages).

**Why it's wrong:** Q2 resolution (per `wiki-entity-restructure-design.md` and PROJECT.md) is hard delete. Stale entity pages create dangling wikilinks in the index. The vault is disposable; orphaned pages are not acceptable.

**Do this instead:** `entity_writer.write_entities` hard-deletes pages for nodes not in the current graph. Callers that need the old stale-tag pattern for non-entity wiki pages continue to use `_add_stale_tag` as before.

### Anti-Pattern 4: Per-entity LLM call for every scan

**What it would look like:** Invoking the scanner LLM for every entity page on every scan, including pages whose graph structure has not changed.

**Why it's wrong:** Cost and latency. The entity writer's deterministic pass is free; LLM calls are expensive.

**Do this instead:** Gate LLM invocations on `needs_narrative`: the entity writer marks a page `needs_narrative: true` in its internal state when (a) the page is newly created, or (b) the graph-derived structural data changed significantly (e.g. a new domain assignment or new test suite). Only those pages enter the LLM fan-out.
